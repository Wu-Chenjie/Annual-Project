# Annual-Project 短期规划书（P0 工程化闭环）

> 基准文档：[deep-research-report.md](./deep-research-report.md)
> 时间起点：**2026-05-08**　｜　里程碑：**2026-06-08**（约 22 个工作日）
> 核心目标：**让仓库从"能跑的研究代码"升级为"可复现、可协作、可答辩的工程平台"**

---

## 一、背景与目标

deep-research-report 将仓库现状定性为"已具备答辩价值，但离公开、长期、可复现实验平台还差工程化闭环"。短期阶段聚焦 5 项 P0 任务，全部围绕**降低上手成本、统一产出格式、锁定运行环境、明确能力边界、收敛仓库结构**。这 5 项完成后，仓库将具备：

- 一条命令完成"环境安装 → 全量测试 → benchmark → 结果汇总"的完整复现链路
- 跨 Python/C++/Web 三条运行线的统一结果 schema 与自动对比能力
- 可重现的锁定依赖环境（Docker / Conda）
- 清晰的三线能力矩阵，消除"文档说有、代码没有"的漂移风险
- 聚焦 UAV 主线的干净仓库边界

## 二、任务分解

### P0-1　一键复现实验入口

**目标**：将分散在 `main.py`、`benchmark.py`、`pytest`、各场景脚本中的入口统一为一个可复现命令。

**现状**：
- 单次仿真：`python main.py`（需手动改 config 预设）
- 批量评测：`python simulations/benchmark.py`
- 测试：`pytest`（入口在 `pytest.ini` 和 `pyproject.toml` 两处声明）
- 场景切换依赖修改 `main.py` 源码中的 `get_config(preset)` 调用

**具体任务**：

| # | 任务 | 说明 |
|---|------|------|
| 1.1 | 统一 CLI 入口 | 在 `main.py` 中接入 `argparse`，支持 `--preset warehouse` 等参数切换，消除改源码切换场景的现状 |
| 1.2 | 全量复现脚本 | 新增 `scripts/reproduce.sh`（Linux/macOS）和 `scripts/reproduce.ps1`（Windows），顺序执行：环境检查 → 全量 pytest → 关键场景仿真 → benchmark → 结果汇总 |
| 1.3 | CI 就绪的 pytest 入口统一 | 删除 `pytest.ini` 与 `pyproject.toml` 中的重复声明，统一到 `pyproject.toml`，添加 `pytest-xdist` 可选并行 |
| 1.4 | 结果目录标准化 | 统一所有输出到 `outputs/<preset>/<timestamp>/` 结构，包含 `sim_result.json`、`trajectory_3d.png`、`error_realtime_3d.png`、`risk_report.json` |
| 1.5 | README 复现章节 | 在 README 添加"快速复现"章节，≤10 行命令即可完成全流程 |

**交付物**：
- `main.py` 支持 `--preset` 参数
- `scripts/reproduce.sh` + `scripts/reproduce.ps1`
- 统一的 `pyproject.toml` 测试配置
- 标准化输出目录结构
- README 复现章节

**验收标准**：
- 新机器执行 `scripts/reproduce.sh` 一键完成全部流程并输出汇总摘要
- 16 个预设场景均可通过 `--preset` 切换运行

**预估工作量**：2–4 天

### P0-2　统一 Benchmark Schema

**目标**：定义 Python/C++/Web 三条运行线共用结果 JSON schema，消除字段不一致。

**现状**：
- Python `benchmark.py` 输出字段：`runtime_s`、`mean_error`、`max_error`、`final_error`、`completed_waypoint_count`
- C++ `benchmark.cpp` 声称输出相同 schema，但文档指出"C++ 日志字段不完整"
- Web 端 `simulate` API 返回 C++ `sim_dynamic_replay` 的输出，字段未明确定义
- 各场景仿真输出的 `sim_result.json` 字段与 benchmark 不同（含 `planned_path`、`executed_path`、`replan_events`、`sensor_logs`、`collision_log`、`fault_log` 等）

**具体任务**：

| # | 任务 | 说明 |
|---|------|------|
| 2.1 | 定义 JSON Schema | 新增 `schemas/benchmark_result.schema.json` 和 `schemas/sim_result.schema.json`，使用 JSON Schema draft-07 |
| 2.2 | Python 侧对齐 | 修改 `benchmark.py` 和场景仿真输出，严格遵循 schema，添加 `validate()` 函数 |
| 2.3 | C++ 侧补齐 | 修改 `benchmark.cpp` 和场景 main，补齐缺失字段，使用 nlohmann/json 输出符合 schema 的 JSON |
| 2.4 | Web 侧适配 | 在 `server.py` 的 `/api/simulate` 响应中包装为符合 schema 的结构 |
| 2.5 | 跨语言对比脚本 | 新增 `scripts/compare_results.py`，接收 Python/C++ 两份结果 JSON，自动对比差异并生成 markdown 报告 |
| 2.6 | Schema 文档 | 在 `docs/` 下新增 `benchmark-schema.md`，说明每个字段的含义、单位和适用范围 |

**交付物**：
- `schemas/benchmark_result.schema.json`
- `schemas/sim_result.schema.json`
- Python/C++/Web 三侧输出对齐
- `scripts/compare_results.py`
- `docs/benchmark-schema.md`

**验收标准**：
- 同一场景 Python 和 C++ 输出可通过 `compare_results.py` 自动对比
- JSON Schema 校验通过（使用 `jsonschema` 库 / `ajv` CLI）

**预估工作量**：3–5 天

### P0-3　Python / C++ / Web 能力对齐矩阵

**现状**：
- 技术文档明确标注了三条线的边界：Python 最完整，C++ 为部署子集，Web 为 `leader_centric` + `illustrative_lateral_offsets`
- 已有 `test_cpp_sync_static.py` 做 C++ 静态同步检查，但仅覆盖文件存在性/编译，不覆盖场景级语义对齐
- 部分模块（如 GNNPlanner、FaultDetector、RiskReport）不确定在 C++ 侧的完整程度

**具体任务**：

| # | 任务 | 说明 |
|---|------|------|
| 3.1 | 功能清单梳理 | 按模块枚举 Python 侧全部功能点（controller_kind × planning_algorithm × safety × fault × report），输出功能清单表 |
| 3.2 | C++ 侧逐项核对 | 检查 `cpp/include/` 和 `cpp/src/` 中每个功能点的实现状态，标注"完整/部分/缺失/不适用" |
| 3.3 | Web 侧逐项核对 | 检查 `web/server.py` 中每条 API 的能力边界，标注前端展示与实际计算的差异 |
| 3.4 | 生成对齐矩阵 | 输出 `docs/capability-matrix.md`，用表格呈现三线 × 功能点的覆盖状态 |
| 3.5 | 场景级回归测试 | 将 `test_cpp_sync_static.py` 扩展为 `test_cross_line_regression.py`，选取 3 个代表场景（basic/warehouse/school_corridor），Python 和 C++ 同时运行，对比轨迹误差 |
| 3.6 | 漂移修复 | 对矩阵中标注"文档声称有但代码缺失"的项，修正文档或补最小实现 |

**交付物**：
- `docs/capability-matrix.md`
- `tests/test_cross_line_regression.py`
- 对齐修复 PR（文档修正 + 最小代码补齐）

**验收标准**：
- 矩阵覆盖所有模块 × 三条线的状态，无"未知"项
- 3 个代表场景的 Python/C++ 跨线回归测试通过

**预估工作量**：4–7 天

### P0-4　环境锁定（Lockfile / Docker / Conda）

**目标**：让任意新机器在 10 分钟内完成可运行环境的搭建，消除"能装但跑不一致"的风险。

**现状**：
- `requirements.txt` 仅声明宽松下界：`cvxpy>=1.4`、`osqp>=0.6`
- 无 lockfile、无 Dockerfile、无 `environment.yml`
- `cvxpy`/`osqp` 求解器栈在不同平台/后端下可能产生数值差异
- Web 端有独立 `requirements.txt`，与主项目依赖未统一
- COLMAP/OpenMVS 路径硬编码（`D:\tools\photogrammetry\...`）

**具体任务**：

| # | 任务 | 说明 |
|---|------|------|
| 4.1 | 生成 lockfile | 在当前可运行环境中执行 `pip freeze` 或 `pip-tools compile`，生成 `requirements.lock.txt`（含精确版本号）和 `requirements-dev.lock.txt` |
| 4.2 | Conda 环境文件 | 编写 `environment.yml`，覆盖 Python 主依赖 + C++ 构建工具链（cmake、ninja），支持 `conda env create -f environment.yml` |
| 4.3 | Dockerfile | 编写 `Dockerfile`（基于 `python:3.10-slim`），安装依赖 + 编译 C++ 子项目，入口为复现脚本。添加 `.dockerignore` |
| 4.4 | 依赖声明统一 | 合并主项目和 Web 项目的依赖声明，`pyproject.toml` 增加 `[project.optional-dependencies]` 的 `web` 组 |
| 4.5 | C++ 依赖说明 | 在 `cpp/README.md` 中列出 C++ 编译依赖（CMake ≥ 3.20、支持 C++20 的编译器），说明 Windows/Linux 构建步骤 |
| 4.6 | 求解器兼容说明 | 在文档中说明 `cvxpy`/`osqp` 在不同后端下的数值差异风险，推荐 SCS 作为跨平台回退 |

**交付物**：
- `requirements.lock.txt` + `requirements-dev.lock.txt`
- `environment.yml`
- `Dockerfile` + `.dockerignore`
- 更新 `pyproject.toml`（web 可选依赖组）
- `cpp/README.md`

**验收标准**：
- `docker build -t uav-sim . && docker run uav-sim scripts/reproduce.sh` 成功输出与本地一致的结果
- `conda env create -f environment.yml` 可成功创建环境并运行全量测试

**预估工作量**：3–6 天
