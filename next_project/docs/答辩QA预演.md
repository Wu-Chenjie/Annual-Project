# 答辩 QA 预演

本文用于答辩前自问自答训练。回答风格尽量口语化，但保留关键术语，适合现场直接复述。

## 一、项目整体类问题

### Q1：你的项目到底做了什么？

**答：**

我的项目是一个室内多无人机编队飞行仿真系统。它解决的是多架四旋翼无人机在复杂室内环境中保持队形、避开障碍物、在线重规划路径，并在无人机发生执行器故障时进行故障检测和拓扑重构的问题。

系统包含四旋翼动力学、编队控制、A*/Hybrid A* 路径规划、FIRI 凸走廊精修、改进 APF 实时避障、Safe/Danger 双模式 GNN 规划、风险自适应重规划和容错拓扑重构。

### Q2：为什么要做多无人机编队，而不是单机？

**答：**

单机只能完成有限任务，而多无人机编队可以覆盖更大范围、提高任务效率和鲁棒性。例如巡检、搜索、仓储扫描、灾害场景探测都可能需要多机协同。

但是多机编队也更难，因为不仅每架无人机要避障，还要保持相对位置，避免队形扫到障碍物。因此这个项目重点研究“编队保持 + 避障 + 在线变化 + 容错”的综合问题。

### Q3：项目的输入和输出分别是什么？

**答：**

输入主要包括：

- 场景配置 preset。
- 地图 JSON。
- 无人机数量和队形参数。
- 航点序列。
- 控制器和规划器参数。
- 是否启用在线重规划、GNN Danger、容错等功能。

输出主要包括：

- leader 和 follower 的轨迹。
- 编队误差。
- 规划路径和执行路径。
- 在线重规划事件。
- 传感器日志。
- 碰撞日志。
- 故障日志。
- 2D、3D 和 HTML 可视化结果。

### Q4：这个项目最核心的技术路线是什么？

**答：**

可以概括为四层：

1. **动力学与控制层**
   用四旋翼动力学模型和 PID/SMC/反步控制实现轨迹跟踪。

2. **规划层**
   用 A*、Hybrid A*、D* Lite、FIRI 生成安全路径。

3. **在线安全层**
   用传感器、WindowReplanner、APF 和 GNN Danger 模式处理动态风险。

4. **鲁棒容错层**
   用故障注入、FaultDetector 和 TopologyGraph 实现故障检测与拓扑重构。

## 二、动力学与控制问题

### Q5：什么是四旋翼动力学？

**答：**

四旋翼动力学描述的是无人机在四个转子推力作用下的位置、速度、姿态和角速度如何随时间变化。简单说，控制器给出推力和力矩，动力学模型根据牛顿-欧拉方程计算下一时刻状态。

项目中状态包括位置、速度、欧拉角和角速度，控制输入包括总推力和三个方向力矩。

### Q6：为什么要同时支持 PID、SMC 和反步控制？

**答：**

不同控制器适合不同验证需求。

PID 简单、直观，适合基础跟踪。SMC 对风扰和模型误差更鲁棒。反步控制更适合非线性系统的分层设计。

项目保留多种控制器，是为了在不同场景下比较稳定性和误差表现，而不是只依赖单一控制方法。

### Q7：leader 和 follower 是怎么协同的？

**答：**

leader 跟踪任务路径。follower 不直接跟踪全局航点，而是跟踪 leader 的当前位置加上队形偏移。

例如 follower 的目标位置是：

```text
target_follower = leader_position + formation_offset
```

这样 leader 负责全局任务，follower 负责保持队形。

## 三、路径规划问题

### Q8：为什么要用 A*？

**答：**

A* 是一种经典图搜索算法，适合在占据栅格地图上找从起点到终点的最短或近似最短路径。它结合了已走代价和到目标的启发距离，所以比盲目搜索更高效。

在本项目里，A* 是基础规划器，适合仓库、会议室等离散地图。

### Q9：为什么还要 Hybrid A*？

**答：**

A* 只在离散格子中搜索，路径可能不够平滑。Hybrid A* 引入航向和运动原语，生成更接近可执行轨迹的路径。

在公司格子间、实验室这类布局复杂、需要更平滑绕行的场景中，Hybrid A* 更合适。

### Q10：D* Lite 的作用是什么？

**答：**

D* Lite 适合环境发生变化后的增量式重规划。普通 A* 每次都从零搜索，动态环境下开销较大。D* Lite 可以复用之前搜索结果，只更新受影响部分。

在本项目里，它是在线重规划的退化层之一，用于传感器发现局部障碍后的快速修正。

### Q11：什么是 FIRI？

**答：**

FIRI 可以理解为围绕原始路径构造一条凸安全走廊，然后把路径点投影到这个走廊里，使路径离障碍物更安全。

本项目实现的是工程化 FIRI：用半空间约束、解析分离平面、内接球 fallback 和 SDF 推离做路径精修，而不是完整复杂的 MVIE 迭代。

### Q12：为什么 FIRI 不是完整论文复现？

**答：**

完整 FIRI 往往需要反复求解凸优化问题，工程复杂度和依赖都比较高。本项目的目标是构建可运行的多无人机仿真系统，所以采用了更轻量的工程化版本。

它保留了“凸走廊约束路径安全”的核心思想，但用更可控的投影和 fallback 保证运行稳定。

## 四、在线重规划问题

### Q13：什么是在线重规划？

**答：**

在线重规划是指无人机在飞行过程中根据传感器或环境变化重新生成局部路径，而不是只依赖仿真开始前算好的路径。

例如传感器发现前方有障碍物，系统会重新规划一段局部绕行路径。

### Q14：为什么不能直接用重规划路径替换原始航点？

**答：**

因为原始航点代表任务目标，而重规划路径只是当前局部窗口里的临时执行路径。如果直接替换，系统可能丢失后续任务点，导致 leader 只走到局部路径末端就停止或偏离任务。

所以项目把 `task_waypoints` 和 `replanned_waypoints` 分开：

- `task_waypoints` 保留任务语义。
- `replanned_waypoints` 只作为局部执行辅助。

### Q15：WindowReplanner 是怎么分层的？

**答：**

它大致有三层：

1. local：滑动窗口局部修正。
2. incremental：用 D* Lite 增量调整。
3. global：必要时重建全局参考。

如果启用 Danger 模式，还会在危险区域优先尝试 GNN 可见图规划。

### Q16：风险自适应重规划有什么意义？

**答：**

固定频率重规划不够智能。安全区域不需要频繁重规划，危险区域则需要更快反应。

项目用 SDF 距离和传感器最近距离估计风险：

- 风险高，重规划间隔缩短。
- 风险低，重规划间隔延长。

这样兼顾安全性和计算效率。

## 五、APF 与安全避障问题

### Q17：APF 是什么？

**答：**

APF 是人工势场法。目标点产生吸引力，障碍物产生斥力，无人机沿合力方向调整运动。

它不是全局规划器，而是实时安全层。即使规划路径有误差，APF 也能在控制层给出避障修正。

### Q18：传统 APF 有什么问题？

**答：**

主要问题是容易陷入局部极小值，也可能在障碍物附近振荡。

项目参考改进 APF 论文，加入目标距离衰减、旋转逃逸、通信约束和编队整体势场，缓解这些问题。

### Q19：为什么不能只用编队整体 APF？

**答：**

因为编队整体 APF 通常以编队重心为代表。如果某个 follower 靠近障碍物，但重心还很安全，那这个 follower 可能感受不到足够斥力。

所以项目保留逐机独立 APF，并把编队整体 APF 作为增强项叠加。

## 六、GNN Danger 问题

### Q20：这里的 GNN 是深度学习吗？

**答：**

不是。这里的 GNN 指论文中的神经动力学活动扩散模型，不需要训练数据集，也没有反向传播。

项目把论文中的稠密栅格神经元活动场工程化为“可见图顶点活动扩散”，用于危险区域的局部路径规划。

### Q21：为什么要有 Safe/Danger 双模式？

**答：**

因为不同风险状态需要不同策略。

安全区域用常规 A* / D* Lite / Hybrid A* 更稳定。危险区域，如窄通道或障碍密集区，普通栅格搜索可能因为障碍膨胀找不到合适路径，这时切换到 GNN 可见图规划。

### Q22：Danger 模式怎么触发？

**答：**

主要看两个指标：

- 传感器最近距离是否低于阈值。
- 当前 SDF 距离是否低于阈值。

触发后进入 Danger 模式。恢复 Safe 模式时使用更高阈值和滞回机制，防止模式频繁抖动。

### Q23：可见图是什么？

**答：**

可见图是把障碍物表面的关键点抽出来，如果两个点之间的直线不穿过障碍物，就连一条边。

这样路径规划就变成在这些可见边上找从起点到终点的路线。

### Q24：为什么说 GNN 是工程化变体？

**答：**

论文原始 GNN 是在稠密栅格神经元上扩散活动值。本项目为了效率，把神经元压缩到可见图顶点上。这样保留了活动扩散和贪心提取路径的思想，但不是逐字复现论文原始形式。

答辩时可以主动说明这是工程改写，避免被问“是否严格复现”时被动。

## 七、容错问题

### Q25：项目如何模拟无人机故障？

**答：**

项目通过转子故障掩码模拟执行器失效。比如某个转子 severity = 0.3，表示该转子只能输出 30% 推力。

这样从机会产生跟踪偏差，系统就可以测试故障检测和重构逻辑。

### Q26：如何检测故障？

**答：**

使用状态异常检测，主要包括：

- 速度突变是否持续超过阈值。
- 实际位置和期望位置偏差是否持续过大。
- 控制量是否长时间接近饱和。

任一规则触发，就认为该 follower 可能故障。

### Q27：拓扑重构怎么做？

**答：**

检测到故障后，系统从候选队形中选择一个更适合剩余健康无人机的拓扑。

选择依据是图论中的代数连通度 λ₂。λ₂ 越大，说明队形图连通性和鲁棒性越好。

### Q28：为什么简单场景默认不启用故障检测？

**答：**

因为故障检测基于阈值，风扰、跟踪误差或路径切换都可能造成短暂异常。如果简单场景默认开启，可能误报故障。

所以项目只在容错相关 preset 中启用，其他场景默认关闭，保证回归稳定。

## 八、论文结合问题

### Q29：四篇论文分别对应项目哪些部分？

**答：**

论文 1 对应改进 APF，包括目标距离衰减、旋转逃逸、通信约束和编队整体势场。

论文 2 对应 Safe/Danger 双模式和 GNN 分层规划，本项目实现为可见图 GNN 变体。

论文 3 对应多智能体强化学习中的优先级思想，本项目没有完整实现 MADDPG，而是借鉴其风险重要性思想，做成风险自适应重规划间隔。

论文 4 对应容错拓扑重构，包括故障注入、故障检测和基于 Laplacian λ₂ 的拓扑选择。

### Q30：项目有没有完全复现论文？

**答：**

没有全部逐字复现。项目更偏工程整合和仿真验证。

例如 GNN 是可见图顶点变体，FIRI 是工程化凸走廊，MADDPG 只借鉴风险调度思想。这样做是为了在一个统一系统中稳定运行多机编队、避障、重规划和容错，而不是单独复现某一篇论文。

### Q31：这种工程化改写合理吗？

**答：**

合理。因为项目目标是综合系统验证，而不是单篇论文复现。工程化改写保留了论文核心思想，并通过测试和可视化验证效果。

同时文档中明确标注了哪些是严格实现，哪些是变体或启发式实现，避免夸大。

## 九、实验和验证问题

### Q32：你怎么证明系统有效？

**答：**

主要从四方面验证：

1. 自动测试：覆盖规划器、FIRI、GNN、重规划语义、容错模块等。
2. 场景运行：basic、warehouse、corridor、laboratory 等多场景。
3. 可视化：查看 3D 轨迹、俯视图、误差曲线、重规划点。
4. 指标统计：平均误差、最大误差、完成航点数量、碰撞日志、故障日志等。

### Q33：可视化图怎么看？

**答：**

重点看：

- leader 是否按任务航点走。
- follower 是否保持队形。
- 轨迹是否穿越障碍物。
- 重规划点是否只在局部出现。
- 故障后是否有检测和重构日志。
- 误差曲线是否收敛或保持在可接受范围内。

### Q34：为什么会有 Python 和 C++ 两套？

**答：**

Python 适合快速实现算法、测试和可视化。C++ 适合后续性能验证和工程部署。

目前 Python 是主线，C++ 是可运行移植子集，已经同步了主要能力，包括 FIRI、GNN Danger 和容错，但事件日志和 Plotly 可视化仍以 Python 更完整。

### Q35：C++ 有没有编译验证？

**答：**

当前机器没有安装 `cmake`、`g++`、`clang++` 或 MSVC `cl`，所以没有完成真实 C++ 编译验证。

但是已经补充了 C++ 静态同步测试，检查关键模块、接口和调用链是否存在。后续安装工具链后，应立即执行 CMake 构建和 C++ 场景运行验证。

## 十、质疑与防守

### Q36：为什么不用深度强化学习直接端到端控制？

**答：**

端到端强化学习需要大量训练数据和安全约束验证，在室内多无人机避障中风险较高。本项目选择可解释的规划和控制方法，便于分析、调参和答辩说明。

论文 3 的强化学习思想主要被用于风险调度启发，而不是直接替代可解释控制链路。

### Q37：为什么不用单一算法解决所有问题？

**答：**

因为多无人机室内飞行是复合问题。全局路径、局部避障、实时控制、故障容错的时间尺度不同。

单一算法很难同时做到全局最优、实时响应、稳定跟踪和容错。所以项目采用分层架构，每层解决自己的问题。

### Q38：GNN 路径一定比 A* 好吗？

**答：**

不一定。GNN 在本项目中不是替代 A* 的全局最优算法，而是 Danger 模式下的局部增强。

如果 GNN 路径不可见、过长或失败，系统会回退到局部规划，保证安全性优先。

### Q39：APF 会不会导致局部极小值？

**答：**

传统 APF 会。本项目通过目标距离衰减、旋转逃逸力、FIRI 精修和全局/局部规划结合来缓解。

APF 不是唯一决策层，而是实时修正层，所以不会单独决定完整路径。

### Q40：拓扑重构的 λ₂ 是精确最优吗？

**答：**

它是在候选队形集合中的最优选择，不是对所有可能拓扑做无限搜索。这样计算更可控，适合实时仿真。

λ₂ 用作连通性和鲁棒性的指标，能合理指导故障后的队形选择。

### Q41：项目最大的局限是什么？

**答：**

主要有四点：

- 当前是仿真系统，还没有真实无人机实验。
- C++ 需要安装工具链后做真实编译和运行验证。
- GNN 和 FIRI 都是工程化变体，不是完整论文级复现。
- 在线重规划还没有完整 MPC 时间化轨迹优化。

## 十二、答辩短句库

### 项目定位

本项目不是单一规划算法，而是一个多无人机编队飞行的综合仿真框架。

### 论文关系

我没有把论文简单堆叠，而是把四篇论文分别映射到安全增强、双模式规划、风险调度和容错重构四个层次。

### GNN 说明

这里的 GNN 不是深度学习训练网络，而是神经动力学活动扩散模型的工程化可见图变体。

### FIRI 说明

FIRI 在项目中用于路径安全走廊精修，目前实现的是工程化版本，重点保证稳定可运行。

### 在线语义

在线重规划只更新局部执行路径，不覆盖原始任务航点，这是保证任务语义正确的关键。

### 容错说明

容错链路包括转子故障注入、状态异常检测和基于拓扑连通性的队形重构。

### 局限说明

目前主要是仿真验证，真实部署还需要 C++ 工具链编译验证、硬件接口和实机安全测试。

## 十三、三分钟答辩稿

各位老师好，我的项目是室内复杂环境下多无人机编队飞行、避障、在线重规划和容错控制仿真系统。

项目采用 leader-follower 编队结构，leader 负责跟踪任务航点，follower 根据队形偏移跟随 leader。底层建立了四旋翼动力学模型，并支持 PID、滑模控制和反步控制，用于保证轨迹跟踪稳定性。

在路径规划方面，系统支持 A*、Hybrid A*、D* Lite 和 FIRI 凸走廊精修。A* 负责基础栅格规划，Hybrid A* 负责更平滑的复杂场景路径，D* Lite 用于局部环境变化后的增量式重规划，FIRI 用于把路径进一步投影到安全走廊中。

在在线避障方面，项目引入了传感器、WindowReplanner、改进 APF 和 Safe/Danger 双模式。安全状态下使用常规分层重规划；危险状态下，根据传感器距离和 SDF 距离触发 Danger 模式，并使用 GNN 可见图变体生成局部绕障路径。这里的 GNN 不是深度学习网络，而是论文中神经动力学活动扩散模型的工程化实现。

在风险调度方面，项目借鉴多智能体强化学习中优先级思想，将其转化为风险自适应重规划间隔。环境风险高时缩短重规划周期，风险低时延长周期。

在容错方面，项目实现了转子故障注入、基于状态异常的故障检测，以及基于 Laplacian 代数连通度 λ₂ 的拓扑重构。这样当某个 follower 故障后，系统可以检测异常并选择更合适的队形继续任务。

项目同时维护 Python 主线实现和 C++ 移植子集。Python 侧用于算法研究、测试和可视化，C++ 侧用于后续性能验证。目前系统已经支持多种场景，包括仓库、走廊、会议室、公司格子间和实验室，并通过轨迹图、误差统计、重规划日志和故障日志进行验证。

我的主要工作可以概括为：构建多层安全规划控制框架、实现 Safe/Danger 双模式 GNN 规划、修正在线任务航点与局部路径语义、加入风险自适应重规划和容错拓扑重构，并整理了完整技术文档和测试验证链路。

## 十四、中期新增高频问答（2026-05-13）

### Q1：你现在的中期目标到底达标了吗？

达标。我的判断依据不是主观完成度，而是四类证据：第一，全量测试当前为 `170 passed, 3 skipped`，跳过项都是构建或环境开关型；第二，C++ 关键目标可以构建；第三，C++ 报告测试 `6 passed`，跨线回归开启后 `3 passed`；第四，中期要求的轨迹指标、编队安全、消融报告和中文报告都有产物路径。

### Q2：中期最能体现创新的点是什么？

不是单独用了某个规划器，而是把“编队安全裕度”从结果检查前移到了规划决策中。leader-only planner 只看领航机路径，我这里增加 formation-aware clearance，考虑从机包络和队形宽度；再通过前瞻窗口提前识别窄通道和急转弯；如果局部窗口误判成死路，再用 RRT escape 生成候选路径。最后通过消融实验对比 leader-only、普通 formation-aware 和 lookahead adaptive 三种模式。

### Q3：为什么后来不把 jerk 平方积分作为主要最优判据？

四旋翼不是简单二阶质点系统。对位置轨迹而言，jerk 可以反映一部分平滑性，但四旋翼控制通常还会关注更高阶导数，尤其 snap 对姿态变化和控制输入平滑更有解释力。所以我保留 `trajectory_jerk_squared_integral` 作为辅助指标，把 `trajectory_snap_squared_integral` 作为更高阶的平滑性代理。这里仍是工程指标，不宣称完成了完整最优控制求解。

### Q4：RRT escape 解决的是什么问题？

它解决的是“局部前瞻窗口看到死路，但全局上有绕行通道”的问题。只沿当前窗口采样时，大角度转弯或假分支容易被判定为不可达；RRT escape 提供一个从当前点到任务目标或窗口端点的候选绕行路径，再通过 clearance gate 接收。报告中会记录 `rrt_escape_attempt`、`rrt_escape_accepted` 和 `rrt_escape_failed`，不是黑箱替换路径。

### Q5：C++ 和 Python 现在一致了吗？

要分层回答。Python 是完整研究主线；C++ 是可运行子集。当前 C++ 已对齐关键 preset、配置字段、编队调控事件、中文报告和跨线 schema 回归，但我不会说 C++ 完整复现了 Python 的全部 RRT* / Informed RRT* 能力。这个边界已经写进 `docs/capability-matrix.md`。

### Q6：你的碰撞减少证据在哪里？

可以看 `outputs/ablation_rrt_dual_channel/summary.csv` 和 `outputs/ablation_formation_maze_stress/summary.csv`。在两个压力场景中，`formation_aware_lookahead_adaptive` 都记录了前瞻阻断和 RRT accepted 事件，同时硬碰撞为 0。这个结论不是靠一张图，而是来自 `formation_adaptation_events`、`summary.csv` 和报告图共同支撑。

### Q7：为什么要把队形变换提前到探测窗口触发？

如果等到 leader 到达窄通道边缘才变形，从机包络可能已经与障碍物冲突，或者局部路径会因为当前队形过宽而被拒绝。提前触发的意义是：在队形还来得及过渡时，先把 diamond / triangle 这类宽队形切成 line 或更窄队形，再进入通道。

### Q8：这个系统现在最大的边界是什么？

第一，仍是仿真系统，还没有实机飞行验证；第二，轨迹优化是轻量工程原型，不是完整 GCOPTER/MINCO；第三，C++ 是可运行子集，Web replay 是展示和回放入口，不是完整多机闭环 benchmark。这些边界我会主动说明，避免把工程原型说成论文级完整复现。
