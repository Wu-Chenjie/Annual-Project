```python
class TrajectoryOptimizer:

        self,
        path: np.ndarray,
        corridor: list[FIRICorridor] | None = None,
        max_vel: float = 2.0,
        max_acc: float = 2.0,
    ) -> TrajectoryResult:
        ...
```

### 4.3 技术路线

第一版采用 **minimum jerk / minimum snap 多项式** 或 **uniform B-spline**，不立刻引入重优化依赖。输入为当前 A*/FIRI 输出路径，输出为带时间参数的连续轨迹。

### 4.4 接入点

- `ObstacleScenarioSimulation._plan_offline()`：离线路径生成后可选调用。
- `WindowReplanner` 输出路径后：在线局部窗口内可选调用。
- `Controller.compute_control(..., target_vel, target_acc ...)`：控制器需扩展前馈接口。

### 4.5 回退策略

```text
FIRI corridor -> trajectory optimization success -> use optimized trajectory
FIRI corridor -> trajectory optimization fail -> use current FIRI refined path
FIRI refined path unsafe -> fallback segment planner
fallback unsafe -> reject path and log clearance_blocked
```

|---|---|

| 轨迹采样点 clearance | 全部通过 SDF 检查 |

```python
# core/drone_params.py
@dataclass
class DroneParams:
    name: str
    mass: float
    inertia: np.ndarray
    arm_length: float
    rotor_kf: float
    rotor_km: float
    motor_tau: float
    omega_max: float
    drag_linear: np.ndarray
    drag_quadratic: np.ndarray
    rotor_drag: np.ndarray

    def __post_init__(self):
        assert self.max_total_thrust > self.mass * 9.81 * 1.3
        assert self.hover_omega < self.omega_max
        assert np.all(np.linalg.eigvals(self.inertia) > 0)

    @property
    def max_total_thrust(self) -> float:
        return 4.0 * self.rotor_kf * self.omega_max ** 2

    @property
    def hover_omega(self) -> float:
        return np.sqrt(self.mass * 9.81 / (4.0 * self.rotor_kf))

# 注册表
DRONE_PROFILES: dict[str, DroneParams] = {
    "default_1kg": DroneParams(...),
    "indoor_micro": DroneParams(...),
    "light_uav": DroneParams(...),
}
```

### 13.4 本轮按审查样板复核结论（2026-05-06）

本节按“问题 → 证据 → 影响 → 测试缺口”的样板记录本轮已完成的代码审查结论。这里仅登记**已在代码中确认**的问题，不把尚未验证的猜测写入缺陷表。

#### R1. C++ `λ₂` 实现错误，拓扑重构选型依据失真

- **问题**：C++ `TopologyGraph::algebraic_connectivity()` 不是在算 Laplacian 的第二小特征值，而是在返回最小节点度。
- **证据**：
  - Python 正确实现：`core/topology.py:228-232` 对 `self._laplacian` 调用 `np.linalg.eigvalsh(...)` 并返回 `eigenvals[1]`。
  - C++ 当前实现：`cpp/src/topology.cpp:191-208` 遍历节点累计 `degree += exp(-(d*d)/4.0)`，最后返回 `min_degree`。
- **影响**：
  - `fault_reconfigure()` 依赖 `algebraic_connectivity()` 选择“最佳”拓扑，当前 C++ 侧会按错误指标选队形。
  - 这不是 Python/C++ 的“工程近似差异”，而是核心语义不一致。
- **测试缺口**：
  - `tests/test_fault_tolerance.py` 目前只验证 Python 侧 `lambda2 > 0`，不足以锁住数值语义。
  - `tests/test_cpp_sync_static.py` 未检查 C++ 是否真的构建 Laplacian 并求第二小特征值。
- **建议**：
  - 将 `cpp/src/topology.cpp` 改为显式构建 Laplacian，再求第二小特征值。
  - 补充一个 Python/C++ 对照测试，至少对固定 offsets 的 `lambda2` 排序结果保持一致。

#### R2. 编队包络仍使用各向同性 `max(||offset||)`，R7 问题未修复

- **问题**：Python/C++ 两侧 `envelope_radius()` 都仍在使用各向同性最大范数，尚未实现分轴包络。
- **证据**：
  - Python：`core/topology.py:136-150`，`max(float(np.linalg.norm(off)) for off in offsets)`。
  - C++：`cpp/src/topology.cpp:167-175`，`const double d = norm(off); if (d > max_dist) ...`。
- **影响**：
  - 线性队形的纵向展开会被错误解释为横向膨胀，正是 `docs/优化反效果归因分析.md` R7 批评的缺陷。
  - 对仓库、走廊、会议室这类窄通道场景，会放大为规划不可通行或过度保守。
- **测试缺口**：
  - `tests/test_obstacle_scenario.py:test_envelope_radius()` 只比较不同队形半径大小，没有验证“线性队形横向包络不应随纵向长度放大”。
  - 当前没有 `envelope_per_axis()` 的单测，也没有 C++ 同步检查。
- **建议**：
  - 新增 `envelope_per_axis()`，返回 `(lateral, longitudinal, vertical)`。
  - `envelope_radius()` 标记 deprecated，仅作为兼容接口保留。
  - 增加单测，直接锁住 line/diamond/v_shape 的三轴包络语义。

#### R3. `_inflate_r()` 仍绑定 `initial_formation`，与运行时队形切换脱节

- **问题**：规划膨胀半径仍按初始队形计算，不随 `current_formation` 更新。
- **证据**：
  - Python：`simulations/obstacle_scenario.py:397-403` 直接读取 `self.config.initial_formation`。
  - C++：`cpp/src/obstacle_scenario.cpp:105-108` 新建 `FormationTopology` 后读取 `config_.initial_formation`。
- **影响**：
  - 即使运行时队形已切换，规划层仍按旧包络做栅格膨胀。
  - 控制层与规划层的编队语义发生脱节，会出现“队形已经变了，但路径仍按旧包络判死”的问题。
- **测试缺口**：
  - 当前没有测试覆盖“切换到 line 后，规划膨胀应随之缩小”。
  - 现有静态同步测试主要关注在线路径接收门禁，没有覆盖队形切换后的膨胀语义。
- **建议**：
  - `_inflate_r()` 改读 `topology.current_formation`。
  - 若改为分轴膨胀，则直接转向 `envelope_per_axis(current_formation)`。
  - 补一个场景级回归：先切换队形，再断言规划膨胀/可通行性随之变化。

#### R4. `auto_shrink()` 仍未接入任何主流程

- **问题**：`core/topology.py` 中 `auto_shrink()` 已有接口，但在线主循环没有调用。
- **证据**：
  - 定义位置：`core/topology.py:178-192`。
  - 在线主循环：`simulations/obstacle_scenario.py:934-1088` 中未见 `auto_shrink(...)` 接入点。
- **影响**：
  - 当前“自动收缩”仍停留在接口层，不是可运行功能。
  - 文档中提到的“窄通道前主动收缩队形”尚未在代码里兑现。
- **测试缺口**：
  - 没有 `tests/test_formation_safety.py`。
  - 也没有任何在线场景测试去验证“通道变窄时触发收缩、通道放宽时恢复”。
- **建议**：
  - 在在线重规划循环中引入通道宽度估计，并实际调用 `auto_shrink()`。
  - 待分轴包络落地后，将 `auto_shrink(channel_width, envelope)` 升级为三轴判定版本。

#### R5. Python/C++ `RangeSensor6` 语义不一致，在线重规划存在跨端漂移风险

- **问题**：Python 和 C++ 虽然都叫 `RangeSensor6`，但并不是同一种测距模型。
- **证据**：
  - Python：`core/sensors.py:46-83` 使用解析求交，对 AABB/Sphere/Cylinder 分别计算精确交点。
  - C++：`cpp/include/sensor.hpp:27-41` 使用固定步长 `max_range / 200` 逐步推进，检测 `field.is_collision(p)`。
- **影响**：
  - 对薄障碍、边界接触、量化误差和最小距离触发条件，两端会给出不同测量结果。
  - 这会进一步影响 `WindowReplanner` 的局部栅格更新、风险评估和重规划触发时机。
- **测试缺口**：
  - 目前只有 Python 侧基础测距测试 `tests/test_obstacle_scenario.py:test_sensor_range()`。
  - 没有任何 Python/C++ 传感器对齐测试，也没有“薄障碍/边界接触”场景测试。
- **建议**：
  - 明确文档口径：这是“实现差异待消除”而不是“允许不同”。
  - 若要做双线一致性，C++ 传感器应向解析求交或等价精度收敛。
  - 补充一组固定障碍物几何体的跨端对照测试。

#### R6. `FormationAPF` / `apf_formation_centroid` 仅停留在类与配置层，未接入运行主流程

- **问题**：项目已提供 `FormationAPF` 类和 `apf_formation_centroid` / `apf_centroid_alpha` / `apf_centroid_beta` 配置项，但当前运行主流程没有实际调用它们。
- **证据**：
  - 实现存在：`core/planning/dual_mode.py:130-187` 定义了 `FormationAPF.compute_formation_avoidance(...)`。
  - 配置存在：`simulations/formation_simulation.py:137-147` 定义了 `apf_formation_centroid`、`apf_centroid_alpha`、`apf_centroid_beta`。
  - 但代码搜索结果显示：仓库内没有任何运行代码调用 `FormationAPF(...)` 或 `compute_formation_avoidance(...)`；命中仅出现在设计文档和类定义本身。
  - `simulations/obstacle_scenario.py:1044-1127` 的实际运行逻辑仍然只调用 `ImprovedArtificialPotentialField.compute_avoidance_acceleration(...)`。
- **影响**：
  - 文档中“编队整体势场已 profile 化接入”的说法与当前运行代码不完全一致。
  - `apf_formation_centroid` 等配置目前属于“无效开关”，即使改值也不会改变仿真行为。
- **测试缺口**：
  - 当前没有测试去验证 `apf_formation_centroid=True` 会改变 leader/follower 的避障加速度。
  - 也没有测试去区分“单机独立 APF”与“编队重心势场叠加”两种行为。
- **建议**：
  - 若保留该能力，应在 `obstacle_scenario.py` 或 `formation_simulation.py` 中显式接入 `FormationAPF`。
  - 若近期不打算接入，应在文档中明确标注为“预留实现，未启用”，避免答辩或审查时误判为已落地功能。

#### R7. C++ 动态回放 `sensor_readings` 为占位值，不是真实传感器输出

- **问题**：动态回放帧里虽然输出了 `sensor_readings` 字段，但当前写入的是固定 `sensor_max_range`，不是实时测距结果。
- **证据**：
  - 字段定义：`cpp/include/dynamic_scenario.hpp:57-67`。
  - 写入位置：`cpp/include/dynamic_scenario.hpp:213-215` 直接将六个方向都赋值为 `config_.sensor_max_range`。
  - 导出位置：`cpp/src/dynamic_main.cpp:494-499` 将 `frame.sensor_readings` 原样写入 JSON。
- **影响**：
  - 前端和审查者会误以为动态回放携带了真实六向传感器数据，但实际上只是占位常量。
  - 这削弱了动态回放作为“在线感知-规划-执行闭环证据”的可信度，也会误导后续基于 replay JSON 的分析。
- **测试缺口**：
  - 当前没有测试校验动态回放 `sensor_readings` 是否随障碍物靠近而变化。
  - 现有静态同步测试只检查了 `clearance_blocked` 和路径接收门禁，没有覆盖 replay 传感器字段语义。
- **建议**：
  - 若要保留该字段，应在动态回放中接入真实 `RangeSensor6` 或等价测距逻辑。
  - 若暂时无法提供真实传感器结果，应删除该字段或改名为显式占位字段，避免语义误导。

#### R8. 本轮未发现新的阻塞级问题：`WindowReplanner` 主链路与 `compute_control` 前馈接口已基本到位

- **说明**：
  - Python `core/planning/replanner.py` 与 C++ `cpp/include/replanner.hpp` 均已具备 local → incremental → global，以及可选 danger 模式的分层结构。
  - `compute_control(..., target_vel, target_acc, target_yaw)` 在 Python 控制器主实现中已统一，障碍物场景主循环也在向 `target_acc` 注入 APF 前馈。
- **保留意见**：
  - 这不等于该两部分“已完全审查完毕”，只是本轮没有发现文档样板级别的新增阻塞问题。
  - 若后续继续深挖，应重点补“接口一致性 + 运行时行为一致性”测试，而不是只看函数签名。

#### R9. C++ APF 仅保留基础版，未对齐 Python 的 profile 化增强能力

- **问题**：Python `ImprovedArtificialPotentialField` 已支持 `adaptive_n_decay`、`k_comm/comm_range`、通信约束力、Rodrigues 旋转逃逸等增强项，并通过 `apf_paper1_profile` 间接加载；C++ 侧 `ImprovedArtificialPotentialField` 仍只有基础参数。
- **证据**：
  - Python 接口：`core/artificial_potential_field.py:54-68` 包含 `adaptive_n_decay`、`k_comm`、`comm_range` 等参数；`simulations/obstacle_scenario.py:95-127` 按 `apf_paper1_profile` 组装 profile。
  - C++ 接口：`cpp/include/artificial_potential_field.hpp:11-25` 仅暴露 `k_rep, r_rep, n_decay, k_inter, s_inter, mu_escape, max_acc`。
  - C++ 实现：`cpp/src/artificial_potential_field.cpp` 中不存在 `adaptive_n_decay`、通信约束力或 `Rodrigues` 旋转势场的对应逻辑。
  - C++ 配置：`cpp/include/obstacle_scenario.hpp:20-61` 与 `cpp/include/config.hpp` 现有预设中均未暴露 `apf_paper1_profile`、`apf_comm_range`、`apf_rotational_escape`、`apf_formation_centroid` 等字段。
- **影响**：
  - 文档若将 Python/C++ 都描述为“APF 三档 profile 已接入”，会高估 C++ 当前能力。
  - 同名场景在 Python/C++ 之间即使 planner 一致，避障行为也可能明显不同，尤其是 `danger` 相关预设。
- **测试缺口**：
  - 现有 `tests/test_apf_enhanced.py` 只覆盖 Python 侧增强 APF，没有任何 C++ 对齐测试。
  - `tests/test_cpp_sync_static.py` 虽检查了 GNN Danger 与 adaptive interval 接线，但没有检查 C++ APF 是否具备 Python 对等 profile 字段与行为。
- **建议**：
  - 若目标是双线一致，应先把 C++ 配置层补齐 `apf_paper1_profile` 及相关参数，再同步实现。
  - 若短期不打算同步，应在技术文档中明确：C++ 仅保留基础 APF，Python 才有增强 profile。

#### R10. C++ 动态回放与普通障碍场景结果口径割裂，缺少任务层/事件层证据字段

- **问题**：普通 C++ 障碍场景 `SimulationResult` 已保留 `task_waypoints`、`replanned_waypoints`、`executed_path`、`fault_log`，但动态回放 `ReplayOutput` 只输出 `waypoints`、`frames`、`dynamic_events` 和 `summary`，没有对应的任务层/执行层字段，也没有 Python 风格的事件日志集合。
- **证据**：
  - 普通障碍场景结果：`cpp/include/formation_simulation.hpp:65-79` 定义 `SimulationResult` 包含 `task_waypoints`、`replanned_waypoints`、`executed_path`、`fault_log`；`cpp/src/obstacle_scenario.cpp:619-624` 实际填充这些字段。
  - 动态回放结果：`cpp/include/dynamic_scenario.hpp:80-97` 的 `ReplayOutput` 只包含 `dynamic_events`、`waypoints`、`frames`、`summaries`。
  - 导出 JSON：`cpp/src/dynamic_main.cpp:468-519` 只写出 `dynamic_events`、`waypoints`、`frames`、`summary`，没有 `task_waypoints`、`replanned_waypoints`、`executed_path`、`fault_log`、`replan_events`、`sensor_logs`、`collision_log`。
- **影响**：
  - 动态回放可以做前端展示，但不足以支撑与 Python 障碍物场景同口径的审查、追责和答辩取证。
  - 同一个“在线重规划是否保持任务层语义”的问题，在普通仿真里可以看 `task_waypoints/replanned_waypoints`，在动态回放里却无法直接复核。
- **测试缺口**：
  - 当前测试只检查动态回放存在 `clearance_blocked` 字段及门禁逻辑，没有检查 replay JSON 是否包含任务层/执行层审查所需字段。
  - 也没有测试去对齐普通 C++ 障碍场景输出与动态回放输出的最小公共语义集。
- **建议**：
  - 为 `ReplayOutput` 增加至少 `task_waypoints`、`executed_path`、`replanned_waypoints` 或等价可追溯字段。
  - 若不打算扩展 replay 输出，应在文档中明确：动态回放只用于展示，不用于完整语义审查。

#### R11. C++ 动态回放中的 `follower_poses` 不是动态仿真结果，而是按 leader 横向平移的占位几何

- **问题**：动态回放帧虽然输出了 `follower_poses`，但这些从机位置不是通过控制器、风场、避障和拓扑切换仿真出来的，而是直接用 leader 位置加固定横向偏移拼接。
- **证据**：
  - 写入位置：`cpp/include/dynamic_scenario.hpp:216-219`，对每个 follower 只计算
    `leader_pose + Vec3{0.0, lateral, 0.0}`。
  - 该逻辑没有调用 `FormationTopology` 当前偏移、没有从机状态积分、没有控制器、没有风场、没有避障。
  - 对比普通障碍场景：`cpp/src/obstacle_scenario.cpp:579-607` 才是真正的 follower 控制与状态更新路径。
- **影响**：
  - 动态回放中的从机轨迹不能作为“编队执行效果”证据使用。
  - 任何关于从机碰撞、编队误差、故障后拓扑恢复、队形收缩效果的观察，若基于该 replay，都会被误导。
- **测试缺口**：
  - 当前没有测试去检查 replay `follower_poses` 是否源于真实从机动力学更新。
  - 也没有测试比较普通障碍场景输出的 follower 轨迹与动态回放中的 follower 轨迹语义是否一致。
- **建议**：
  - 若要把动态回放作为编队级证据，必须接入真实 follower 状态演化。
  - 若短期只做 leader 级回放，应在输出或前端中明确标注 `follower_poses` 为示意值。

#### R12. C++ 动态回放未执行普通障碍场景中的队形切换、故障注入/检测/重构语义

- **问题**：动态回放只推进 leader 路径与动态障碍事件，没有复用普通障碍场景中的 `formation_schedule`、故障注入、故障检测和拓扑重构逻辑。
- **证据**：
  - 动态回放主循环：`cpp/include/dynamic_scenario.hpp:188-299` 仅维护 `leader_pose`、`active_path`、`task_goal_idx` 与动态障碍事件。
  - 搜索 `dynamic_scenario.hpp`，未见 `formation_schedule`、`fault_injection_enabled`、`fault_detection_enabled`、`fault_reconfig_enabled`、`topology` 等主流程接入点。
  - 对比普通障碍场景：`cpp/src/obstacle_scenario.cpp:435-447` 会建立 `FaultDetector`，`453-617` 中包含故障注入、检测、`fault_reconfigure(...)` 和 follower 状态更新。
- **影响**：
  - 动态回放虽然能展示“动态障碍 + 路径门禁”，但不能代表完整在线编队仿真。
  - 文档若把 dynamic replay 当成容错、队形切换或 follower 安全的验证载体，会高估其语义覆盖范围。
- **测试缺口**：
  - 当前没有测试验证 dynamic replay 是否支持 `formation_schedule` 或故障事件。
  - 也没有测试保证普通障碍场景与 dynamic replay 在共享配置下至少覆盖同一组关键语义。
- **建议**：
  - 明确把 dynamic replay 定位为“leader-centric 动态障碍回放器”，不要与完整障碍场景仿真混用。
  - 若要用于更完整的验证，需要把队形切换和故障链路最少同步到事件层或结果层。

#### R13. C++ 动态回放 `summary` 存在统计口径误导，不能直接当作“各 planner 独立对比结果”

- **问题**：dynamic replay 的 `summary` 看起来像是在比较 `dstar_lite`、`astar`、`hybrid_astar` 三个 planner 的独立表现，但内部统计并不是三条独立执行轨迹。
- **证据**：
  - 每帧会对所有 planner 都调用一次 `run_one_planner(...)`，并且 `result.replan_triggered = true` 被无条件置真：`cpp/include/dynamic_scenario.hpp:333-363`。
  - `replans_[planner]` 在每帧都会累加：`cpp/include/dynamic_scenario.hpp:235-239`，因此 `summary.total_replans` 实际更接近“该 planner 被评估的帧数”，不是条件触发式重规划次数。
  - leader 真正执行的只有一条共享轨迹：每帧只选 `selected_path`，优先 `dstar_lite`，其余 planner 并未各自推进独立状态：`cpp/include/dynamic_scenario.hpp:224-246`、`253-279`。
  - `summary.final_dist_to_goal = norm(final_pose - goal)` 对所有 planner 复用同一个 `final_pose`：`cpp/include/dynamic_scenario.hpp:646-674`。
  - `phase_distribution` 只在 `planner == "dstar_lite"` 时输出，但其统计来源 `phase_counts_` 是所有 planner 共享累加：`cpp/include/dynamic_scenario.hpp:665-671`。
  - 前端会把这些值直接渲染成“重规划次数 / 终点距离 / 路径长度”等指标，容易被看成严格横向对比：`web/dynamic_replay.html:3540-3554`。
- **影响**：
  - 当前 replay `summary` 适合做“每帧规划开销观察”，不适合被解释为“各 planner 在同一动态场景下各自完整跑完的最终成绩”。
  - 若答辩或文档把这组 summary 直接当 benchmark，对比结论会失真。
- **测试缺口**：
  - 当前没有测试验证 `total_replans` 的定义是否真是“重规划触发次数”。
  - 也没有测试验证不同 planner 的 `final_dist_to_goal` 是否来源于各自独立执行轨迹。
  - `test_cpp_sync_static.py` 对 replay summary 完全没有语义守门。
- **建议**：
  - 若要保留 summary，应把字段名改成更准确的“planner_eval_count”或“frame_eval_count”等。
  - 若要做真正的 planner 对比，应为每个 planner 维护独立执行状态与独立 `final_pose`。
  - 至少要在前端和文档中明确：当前 summary 不是严格 benchmark，只是共享 leader 轨迹下的逐帧规划统计。

#### R14. 多份文档对 APF profile 与 dynamic replay 的表述存在高估实现风险，需统一纠偏

- **问题**：当前部分文档把一些“仅 Python 主线成立”或“dynamic replay 仅部分成立”的能力写成了项目整体已实现，容易让读者默认 Python/C++/Web 三端都已等价落地。
- **证据**：
  - `docs/项目总改进规划.md:136-137` 写成“已接入 `off / conservative / aggressive` 三档；默认只启用低风险项；已由在线场景和障碍规划回归覆盖”，但代码审查已确认：
    - Python 有 `apf_paper1_profile` 组装逻辑；
    - C++ 没有对应 profile 配置与增强实现（见 R9）；
    - `FormationAPF` / `apf_formation_centroid` 也未接入运行主流程（见 R6）。
  - `docs/项目总改进规划.md:260` 将“编队势场、通信约束”写成“已通过 `apf_paper1_profile` 分档接入”，这对 C++ 侧不成立，对 Python 侧也不完全成立，因为编队重心势场仍未接主流程。
  - `docs/技术文档.md:63` 用“当前 C++ 已覆盖”总述 dynamic replay，与普通在线仿真并列，容易让读者误以为 dynamic replay 也具备 follower 动力学、队形切换、故障链路和真实传感器输出；但 R7、R10、R11、R12、R13 已证明它只是部分语义成立。
  - `docs/技术文档.md:1194-1200` 直接介绍 `apf_paper1_profile` 三档，但没有显式限定这是 Python 主线能力。
  - `docs/答辩QA续写_项目实战版.md:120-126` 虽然谨慎地只对齐“安全门禁语义”，但若脱离上下文单看表格，也容易让人误读为 Python 主线与 C++ 动态回放在整体执行层上等价。
- **影响**：
  - 文档口径一旦高估实现，会直接影响答辩表述、代码审查预期和后续对外汇报可信度。
  - 尤其是 dynamic replay，如果不说明它是 `leader-centric` 且包含占位字段，老师或评审会自然把它当成完整在线编队回放器。
- **测试缺口**：
  - 文档层目前没有任何自动一致性检查去防止“设计文档状态”与“当前实现状态”漂移。
  - 这类问题只能靠人工复核，因此需要在审查结论中显式登记。
- **建议**：
  - 后续统一采用三段口径：
    1. **Python 主线已运行**；
    2. **C++ 在线仿真已同步的子集**；
    3. **dynamic replay 仅用于展示或部分安全语义验证**。
  - 对 `apf_paper1_profile` 明确写成“Python 主线已接入；C++ 暂未对齐”。
  - 对 dynamic replay 明确写成“具备路径接收门禁和执行前胶囊体刹车，但不等于完整编队仿真复现”。

### 13.5 测试覆盖缺口总评（对应 R1-R12）

本节不再新增功能缺陷，而是总结这些缺陷为什么能长期存在。结论很明确：当前测试对“Python 主线行为”覆盖相对较好，但对“Python/C++ 语义一致性”和“dynamic replay 是否只是展示壳”覆盖明显不足。

#### T1. 只有 Python 行为测试，没有 C++ 对齐测试

- **涉及问题**：R1、R2、R4、R5、R6、R9。
- **典型表现**：
  - `tests/test_fault_tolerance.py:72-91` 只验证 Python `ToplogyGraph` 的 `lambda2 > 0` 和 `best_reconfig_topology()` 可返回结果，没有 C++ `λ₂` 数值语义测试，因此 R1 能漏过。
  - `tests/test_obstacle_scenario.py:102-109` 只做 Python `envelope_radius()` 大小关系检查，没有分轴包络测试，更没有 C++ 对齐测试，因此 R2/R4 长期无守门。
  - `tests/test_apf_enhanced.py` 全部是 Python 增强 APF 单测，C++ 完全没有对应对齐测试，因此 R9 只能靠人工审查发现。
  - Python 侧 `RangeSensor6` 只有基础测距单测，没有与 C++ 传感器的对照场景，因此 R5 也没有自动化守门。
- **结论**：
  - 只要问题发生在 “C++ 实现偏离 Python 主线” 这一层，现有测试体系基本拦不住。

#### T2. 只有静态字符串检查，没有运行时行为测试

- **涉及问题**：R7、R10、R11、R12，部分覆盖 R9。
- **典型表现**：
  - `tests/test_cpp_sync_static.py` 的价值主要是“确认代码里有这个入口/字段/函数名”，例如检查 `task_waypoints_`、`path_is_clearance_safe`、`fault_log` 等是否存在。
  - 但它不会发现：
    - `sensor_readings` 实际上是固定常量而不是真实测距（R7）。
    - `ReplayOutput` 虽然有 `frames`，但没有 `task_waypoints/replanned_waypoints/executed_path`（R10）。
    - `follower_poses` 只是 `leader_pose + lateral` 的示意值（R11）。
    - dynamic replay 根本没有跑 `formation_schedule` / 故障链路（R12）。
- **结论**：
  - 现有 C++ 静态同步测试对“有没有这个词”敏感，对“这个词是不是假的占位实现”不敏感。

#### T3. 完全没有测试覆盖的问题

- **涉及问题**：R3、R4、R6、R7、R10、R11、R12 的关键行为面。
- **当前空白点**：
  - 没有测试验证 `_inflate_r()` 会随 `current_formation` 变化（R3）。
  - 没有测试验证 `auto_shrink()` 在在线循环中真的被触发（R4）。
  - 没有测试验证 `apf_formation_centroid=True` 会改变仿真行为（R6）。
  - 没有测试验证 dynamic replay 的 `sensor_readings` 会随障碍接近变化（R7）。
  - 没有测试验证 replay JSON 是否包含任务层/执行层审查所需字段（R10）。
  - 没有测试验证 replay `follower_poses` 是否来源于真实 follower 状态更新（R11）。
  - 没有测试验证 dynamic replay 是否支持 `formation_schedule`、故障注入、故障检测和拓扑重构（R12）。
- **结论**：
  - 这些点目前不是“测试弱”，而是“根本没有守门”。

#### T4. 现有测试的强项与边界

- **强项**：
  - Python 主线的任务航点语义、在线路径接收门禁、故障注入日志、APF 增强数学性质，已有一定覆盖。
  - `tests/test_obstacle_scenario.py:356-365` 这类测试能抓住任务层与执行层混淆。
  - `tests/test_apf_enhanced.py` 对 Python APF 的增强项做了直接数学性质守门。
- **边界**：
  - 这些测试强在“Python 行为正确性”，弱在“C++ 对齐”和“replay 真实性”。
  - 因此不能因为 Python 测试通过，就推导 C++ / Web dynamic replay 的语义也正确。

#### T5. 建议补测顺序

按投入产出比，建议后续优先补以下 6 组测试：

1. **C++ λ₂ 对齐测试**：
   固定一组 offsets，断言 Python/C++ `algebraic_connectivity` 排序一致。

2. **编队包络与膨胀语义测试**：
   覆盖 `current_formation` 驱动的 `_inflate_r()`，以及 line 队形横向包络不被纵向长度放大。

3. **dynamic replay 真实性测试**：
   断言 `sensor_readings` 不是常量占位；若 follower 输出存在，则断言不是简单 `leader+y_offset` 占位。
