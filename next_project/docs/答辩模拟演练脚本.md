# 答辩模拟演练脚本

> 用途：把 `严厉评委攻防手册.md` 里的问题变成可实际演练的答辩流程。建议找同学按评委角色朗读问题，答辩人限时回答。回答时只允许看“证据路径”，不允许照读长答案。

---

## 1. 演练规则

### 1.1 时间安排

| 阶段 | 时间 | 目标 |
|---|---:|---|
| 开场陈述 | 3 分钟 | 讲清项目定位、主线创新、当前证据 |
| 第一轮追问 | 5 分钟 | 测创新性和理论边界 |
| 第二轮追问 | 5 分钟 | 测工程实现和代码定位 |
| 第三轮追问 | 5 分钟 | 测实验可信度和 C++/Python 边界 |
| 收束回答 | 1 分钟 | 主动承认边界并强调中期达成 |

### 1.2 评委角色

| 角色 | 关注点 | 提问风格 |
|---|---|---|
| 主席评委 | 项目价值、工作量、达标情况 | 打断式、压缩时间 |
| 理论评委 | 数学定义、安全证明、轨迹优化 | 抓概念漏洞 |
| 工程评委 | C++、Web、配置、报告链路 | 要求代码路径 |
| 实验评委 | 消融、地图、样本量、指标可信度 | 质疑泛化 |
| 代码评委 | 文件定位、字段来源、测试覆盖 | 现场追字段 |

### 1.3 回答硬规则

- 每个回答先给结论，再解释。
- 超过 90 秒必须主动收束。
- 不确定时说“我按数据链路定位”，不要编行号。
- 被问到边界时先承认，再说明当前已完成部分。
- 不把 quick 测试说成 full 统计实验。

## 2. 开场 3 分钟脚本

### 2.1 标准版本

各位老师好，我的项目是面向室内复杂环境的多无人机编队飞行仿真与报告系统。它解决的问题不是单机路径规划，而是多机编队中 leader 安全不一定代表 follower 安全的问题。

系统底层包含四旋翼动力学、编队控制、路径规划、在线重规划、避障、故障检测和报告输出。中期阶段我重点补齐了一条主线：把从机包络和队形安全裕度纳入规划决策，也就是 formation-aware clearance；再通过前瞻窗口，在进入狭窄通道或大角度转弯前提前触发队形变换；当局部窗口误判前方为死路时，引入 RRT escape 作为候选绕行路径。

这条主线不是只看可视化效果，而是有消融实验支撑。当前可以对比 leader-only、formation-aware adaptive 和 formation-aware lookahead adaptive 三种模式，并输出 CSV、Markdown、图片和事件日志。报告会记录路径参数、飞行参数、误差、碰撞、航点到达、规划耗时、队形调整和 RRT escape attempt/accepted/failed。

当前中期目标已经按工程验收口径达成：Python 主线具备完整实验闭环，C++ 侧作为可运行子集已对齐关键预设、报告字段、障碍物建模和编队调控元数据。边界是：这仍是仿真系统，不是实机安全证明；C++ 不宣称完整等价 Python；轨迹优化当前是工程原型，不等同完整 GCOPTER/MINCO。

### 2.2 被打断时的 30 秒压缩版

如果只能讲 30 秒，我会概括为：本项目把多无人机编队中“leader 安全不代表 follower 安全”的问题转化成可计算、可记录、可消融的仿真链路。核心是 formation-aware clearance、前瞻队形调整和 RRT escape。当前已有两个压力场景消融报告、中文结果报告、C++ 可运行子集和测试记录支撑。边界是仿真验证，不是实机安全证明。

## 3. 第一轮：主席评委追问

### Q1：你别讲这么多模块，直接说你最核心的贡献是什么？

**限时回答：45 秒。**

我的核心贡献是把编队安全裕度从执行后检查前移到规划决策中。普通 leader-only planner 只保证领航机不碰撞，但 follower 有队形包络，在窄通道和转弯处可能擦碰。我把从机包络、通道宽度和安全裕度纳入 clearance gate，并结合前瞻窗口和 RRT escape，让系统能提前变队形、记录阻塞原因，并通过消融报告验证。

**追问：这不就是调大安全半径吗？**

不是统一调大半径。它与当前队形相关，宽队形和窄队形的 envelope 不同；同时它会触发队形调整和事件记录，而不是静态膨胀障碍物。调参改变的是数值，这里改变的是规划接受逻辑。

### Q2：你说中期达成，最硬的证据是什么？

**限时回答：60 秒。**

最硬的证据有三类。第一是测试和构建记录：中期验收记录中记录了全量 pytest、C++ 关键目标构建、C++ 报告测试和跨线回归。第二是实验产物：`outputs/ablation_rrt_dual_channel` 和 `outputs/ablation_formation_maze_stress` 下有 summary 和 report，记录前瞻阻塞、RRT escape 和硬碰撞。第三是文档一致性：能力矩阵、技术文档和答辩索引都明确了 Python/C++/Web 的能力边界。

**追问：如果现场复现失败怎么办？**

我会先区分是环境问题、构建问题还是算法问题。现场不硬说通过，而是用最近一次验证命令和已有输出目录说明当前证据，同时定位失败发生在运行环境、C++ 构建还是 Python 实验流程。

### Q3：这个项目最大的短板是什么？

**限时回答：45 秒。**

最大短板是还没有实机验证和形式化安全证明。当前是仿真验证平台，能证明工程链路可运行、事件可追踪、消融可复现，但不能直接宣称实机安全。后续需要补随机地图多 seed、连续时间安全约束、硬件噪声和实机闭环测试。

## 4. 第二轮：理论评委追问

### Q4：你的 formation-aware clearance 能写成数学形式吗？

**限时回答：90 秒。**

可以。leader-only 主要检查 leader 路径到障碍物的距离是否满足安全裕度：

```text
d(p_leader, obstacle) >= r_uav + margin
```

formation-aware clearance 把从机队形包络纳入判定：

```text
d(p_leader, obstacle) >= r_uav + margin + R_formation
```

其中 `R_formation` 来自当前队形下 follower 相对 leader 的最大包络半径或横向宽度。它表达的是：即使 leader 中心线可通过，如果 follower envelope 与障碍物冲突，这条路径也不能被编队接受。

**追问：这个能证明连续时间安全吗？**

不能。当前是离散路径和工程裕度判定，不是连续时间可达性证明。严格证明需要控制屏障函数、可达集分析或连续走廊约束。我的中期贡献是把这个安全判定模块化、可测试、可消融。

### Q5：为什么不用 MPC 或 CBF？

**限时回答：60 秒。**

MPC 和 CBF 更适合做严格安全控制，但中期目标是先完成可运行、可解释、可报告的编队仿真闭环。formation-aware clearance 计算轻、可嵌入现有规划器、能直接输出事件原因，也更适合做消融。后续如果深化理论，可以把当前 clearance gate 替换或扩展为 CBF/MPC 约束。

### Q6：你从 jerk 改成 snap，是不是说明之前理论错了？

**限时回答：60 秒。**

这是对指标的修正。最初 jerk 可作为路径平滑性指标，但四旋翼不是简单二阶质点系统，轨迹到姿态和控制输入通常涉及更高阶导数，minimum snap 在四旋翼轨迹生成中更常见。因此我保留 jerk 作为辅助观察量，把 snap 平方积分作为更合适的高阶平滑代理。但我不宣称已经完成完整 minimum snap 最优控制求解。

## 5. 第三轮：工程评委追问

### Q7：Python 和 C++ 到底哪里一致，哪里不一致？

**限时回答：90 秒。**

一致的是中期报告所需的工程接口和关键语义：C++ 支持关键 preset、路径和飞行参数输出、碰撞/航点/重规划事件、障碍物建模、编队调控元数据，并能通过 `experiments.run_cpp_report` 生成中文报告。

不一致的是完整算法能力：Python 是研究主线，包含更完整的在线仿真和消融链路；C++ 是可运行子集，不宣称完整复现 Python 的全部 RRT*/Informed RRT*、完整 APF profile 和从机动力学细节。

**追问：这是不是偷换“对齐”的含义？**

不是。我会明确说“报告字段和中期场景语义对齐”，不说“所有算法完全等价”。能力矩阵里也按 Python、C++、Web 分层标注了边界。

### Q8：C++ 输出字段会不会是假的？

**限时回答：60 秒。**

要按字段分层。路径、飞行参数、碰撞、航点和规划耗时来自仿真输出或报告计算；编队调控元数据用于对齐报告语义；dynamic replay 中某些 follower 展示字段不能作为完整多机动力学证据。我的文档已经把这些边界写入能力矩阵和后续实施映射。

### Q9：报告生成会不会影响仿真行为？

**限时回答：45 秒。**

不会作为设计目标影响。数据流是单向的：仿真生成 `sim_result.json` 和事件日志，metrics extractor 提取指标，report writer 渲染 Markdown 和图片。报告生成器不应反向进入控制或规划决策。如果发现反向依赖，那就是需要修正的工程设计问题。

## 6. 第四轮：实验评委追问

### Q10：两个压力地图能说明泛化吗？

**限时回答：60 秒。**

不能说明统计泛化。两个压力地图用于机制验证：双通道地图验证局部窗口假死路和 RRT escape；迷宫压力地图验证狭窄通道、大角度转弯和前瞻队形调整。泛化需要后续随机地图、多 seed、更长仿真时间和障碍密度扫描。

### Q11：硬碰撞为 0 会不会只是阈值设置宽松？

**限时回答：75 秒。**

硬碰撞为 0 只是一个结果，不单独作为全部证据。报告同时记录 clearance blocked、formation adaptation events、RRT escape、路径长度、速度、加速度和误差。如果阈值过松，可能会在 clearance 或误差指标中暴露。后续还需要做阈值敏感性分析。

### Q12：planner comparison 里的“最优”是什么意思？

**限时回答：45 秒。**

不是数学全局最优，而是在当前预设、候选规划模式和指标权重下的相对最优。报告会解释为什么综合评分更好，例如碰撞更少、路径更短、耗时更低、平滑性指标更好。严格全局最优不在当前中期主张范围内。

## 7. 第五轮：代码评委追问

### Q13：现场定位 formation-aware clearance 的代码。

**限时回答：45 秒。**

策略模块在 `core/formation_adaptation.py`。在线仿真调用和事件记录在 `simulations/obstacle_scenario.py`。消融开关在 `experiments/ablation.py`。指标提取在 `experiments/metrics_extractor.py`。报告展示在 `experiments/result_reporter.py` 和 `experiments/report_writer.py`。

### Q14：RRT escape 的事件从哪里进报告？

**限时回答：60 秒。**

事件先在在线仿真中生成，主要看 `simulations/obstacle_scenario.py` 的重规划和前瞻逻辑；随后 `experiments/metrics_extractor.py` 提取 `rrt_escape_attempt_count`、`rrt_escape_accepted_count`、`rrt_escape_failed_count` 等字段；最后报告生成器把这些指标写入 Markdown 和图表。

### Q15：如果老师要求你说某个字段具体哪一行生成怎么办？

**限时回答：30 秒。**

我不凭记忆编行号。我会现场搜索字段名，然后按“仿真生成、指标提取、报告渲染”三层定位。这样比背行号可靠。

## 10. 30 分钟训练法

### 第 1 轮：只练开场

连续讲 3 次，每次控制在 3 分钟以内。录音后检查是否出现以下危险词：

- “完全证明”
- “全局最优”
- “C++ 完全一致”
- “所有问题都解决”
- “可以直接上真机”

### 第 2 轮：只练狠问题

随机抽 10 个问题，每题限时 60 秒。回答必须包含：

- 结论
- 边界
- 证据路径

### 第 3 轮：只练代码定位

让同学随机问字段或模块，例如：

- `formation_adaptation_events`
- `trajectory_snap_squared_integral`
- `rrt_escape_accepted_count`
- `cpp_report.md`
- `leader_only_planner`

答辩人必须说出数据链路，不要求背行号。
