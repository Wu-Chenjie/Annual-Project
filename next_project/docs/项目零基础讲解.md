# 项目零基础讲解

本文面向完全不了解无人机、路径规划和控制的人，按“问题是什么 -> 项目怎么拆 -> 每个模块做什么 -> 论文怎么用 -> 运行结果怎么看”的顺序讲解本项目。

## 1. 一句话理解项目

这个项目研究的是：**多架四旋翼无人机在室内复杂环境中保持编队飞行，同时避开障碍物，并在传感器发现新障碍或无人机发生故障时自动重规划和调整队形。**

可以把它想成一个“空中小队”：

- 有一架 leader 负责沿任务航点前进。
- 多架 follower 跟着 leader，保持 V 形、菱形、直线等队形。
- 房间里有货架、墙、柱子、桌椅、实验台等障碍物。
- 小队既要到达目标，又不能撞障碍物，也不能互相撞。
- 如果环境变危险，就要更频繁规划；如果某架从机故障，就要检测并重构队形。

## 2. 为什么这个问题难

单架无人机飞到目标点已经需要控制位置、速度、姿态和推力。多架无人机一起飞时，还多了几个难点：

1. **编队保持**
   每个 follower 不是随便飞，而是要站在 leader 的某个相对位置上。

2. **避障**
   leader 要避障，follower 也要避障。即使 leader 没撞，follower 的队形偏移可能会扫到障碍物。

3. **在线变化**
   离线规划好的路径不一定一直安全。传感器发现局部障碍后，需要重新规划。

4. **局部危险区**
   在窄通道或障碍密集区，普通 A* 可能因为膨胀层太保守而找不到好路径，所以引入 Safe/Danger 双模式。

5. **故障容错**
   某个转子推力下降后，从机会偏离队形。系统要检测异常，并选择新的拓扑或队形继续任务。

## 4. 从一次仿真看完整流程

运行一个场景时，大致发生这些事：

1. `main.py` 读取命令行参数。
2. `config.py` 根据 preset 生成 `SimulationConfig`。
3. 如果没有障碍物，使用 `FormationSimulation`。
4. 如果有障碍物，使用 `ObstacleScenarioSimulation`。
5. 地图 JSON 被加载为障碍物集合。
6. 障碍物被转换为占据栅格。
7. 规划器生成初始路径。
8. leader 跟踪路径，follower 跟踪 leader 的相对队形。
9. APF 实时给 leader 和 follower 增加避障加速度。
10. 在线模式下，传感器每隔一段时间触发 `WindowReplanner`。
11. 如果进入 Danger 模式，切换到 GNN 可见图规划。
12. 如果检测到故障，触发拓扑重构。
13. 仿真结束后输出轨迹、误差、重规划事件和可视化图。

## 5. 四旋翼动力学：无人机怎么“动起来”

四旋翼有四个转子。每个转子的转速决定推力，四个推力组合起来产生：

- 总升力：让无人机上升或下降。
- 滚转力矩：让无人机左右倾斜。
- 俯仰力矩：让无人机前后倾斜。
- 偏航力矩：让无人机绕竖直轴旋转。

项目里 `core/drone.py` 和 `cpp/src/drone.cpp` 都实现了四旋翼状态更新。状态主要包含：

- 位置：`x, y, z`
- 速度：`vx, vy, vz`
- 姿态：roll, pitch, yaw
- 角速度：`p, q, r`

控制器不会直接说“飞到这里”，而是计算出推力和力矩，再由动力学积分得到下一时刻状态。

## 6. 控制器：怎样让无人机跟踪目标

项目里有三类控制思想：

1. **PID 控制**
   根据位置误差、速度误差和误差变化来修正控制量。优点是简单稳定，适合基础场景。

2. **滑模控制 SMC**
   对扰动更鲁棒，适合有风场或不确定性的场景。它会把系统状态拉回设计好的滑模面。

3. **反步控制 Backstepping**
   用分层构造的方式，从位置目标逐步推导出姿态和推力控制，适合非线性系统。

在答辩时可以这样解释：

> PID 负责基础跟踪，SMC 提高抗扰性，反步控制用于非线性四旋翼系统的更稳定跟踪。项目支持开关组合，便于对比。

## 7. 编队拓扑：多架无人机怎么排队

leader 是参考点，follower 的目标位置由 leader 位置加上相对偏移得到。

例如三架 follower 的菱形队形可以理解为：

```text
        follower 2

leader

        follower 1
   follower 3
```

项目支持：

- `v_shape`
- `diamond`
- `line`
- `triangle`
- `custom`

`core/topology.py` 和 `cpp/include/topology.hpp` 负责队形偏移计算、平滑切换和故障后的重构。

队形切换不是瞬间跳变，而是用余弦插值平滑过渡，避免 follower 突然收到很大的目标跳变。

## 8. 障碍物地图：环境怎么表示

地图在 `maps/*.json` 中。障碍物类型主要有：

- AABB：长方体障碍物，如墙、货架、桌子。
- Sphere：球形障碍物。
- Cylinder：圆柱，如柱子、圆桌腿、设备柱。

仿真时会把这些障碍物转成占据栅格：

- 0 表示可通行。
- 1 表示被障碍物占据。
- 膨胀后的格子表示“虽然没有实体，但队形和安全距离可能会碰到”。

这样规划器就能在三维网格中搜索安全路径。

## 9. 路径规划：怎么找到一条路

项目中有多种规划器。

### 9.1 A*

A* 是最基础的图搜索。它在网格中从起点搜索到终点，同时用启发函数估计“离目标还有多远”。

优点：

- 稳定。
- 易解释。
- 适合离散栅格地图。

缺点：

- 网格细时计算量大。
- 不天然考虑运动学平滑。

### 9.2 Hybrid A*

Hybrid A* 在搜索时加入航向和运动原语，更适合需要平滑路径的场景。

在项目中，它用于公司格子间、实验室等有复杂布局的场景。

### 9.3 D* Lite

D* Lite 用于在线环境变化。它不是每次都从零规划，而是尝试增量更新已有路径。

适合传感器发现局部新障碍时快速修正。

### 9.4 Informed RRT*

RRT* 是采样式规划，Informed RRT* 在已有解之后只在更有希望的区域采样，提高效率。

项目中它作为全局重建层的思想来源，和在线重规划框架一起使用。

## 10. 在线重规划：为什么不能直接覆盖原始航点

这是项目里一个很关键的设计点。

原始任务航点是“这趟任务必须经过的目标点”。在线重规划路径只是“当前窗口内为了避开局部障碍而临时生成的子路径”。

如果用局部路径直接覆盖任务航点，会发生问题：

- leader 可能只走到窗口末端就停下。
- 原本必须经过的任务点被丢失。
- 可视化中会看起来像没有按任务执行。

所以现在项目采用分层语义：

- `task_waypoints`：原始任务航点，不被在线重规划覆盖。
- `replanned_waypoints`：重规划产生的局部执行点。
- `executed_path`：leader 实际走过的轨迹。
- `planned_path`：离线规划参考路径。

在线重规划只服务当前任务航点，生成局部路径后再拼回当前任务目标。

## 11. 改进 APF：实时安全层

APF 是人工势场法。可以把目标点看成“吸引力”，障碍物看成“斥力”。

传统 APF 有两个问题：

1. 容易陷入局部极小值。
2. 在密集障碍区可能左右摇摆。

项目结合论文 1 做了增强：

- 目标距离衰减：离目标近时斥力影响方式变化。
- 旋转逃逸力：遇到局部困境时增加切向绕行趋势。
- 通信约束力：避免编队成员距离过远。
- 编队整体势场：让整个队形的安全性更一致。

但项目没有让整体 APF 替代单机 APF。原因是：如果只看编队重心，靠近障碍物的 follower 可能感受不到危险。因此项目采用“逐机独立 APF + 编队整体增强”的叠加策略。

## 12. FIRI 凸走廊：让路径更安全

FIRI 可以理解为：围绕路径段构造一条“安全走廊”，再把路径点投影到这个走廊内部。

项目里的 FIRI 是工程化版本，不是完整论文级 MVIE 迭代。它做了：

- 为路径段构造半空间约束。
- 用障碍物解析几何生成分离平面。
- 用内接球 fallback 代替复杂优化失败情况。
- 对不安全点进行 SDF 推离。

作用：

- 减少路径贴障碍物。
- 让离线和在线路径更稳定。
- 为未来连续轨迹优化打基础。

## 13. Safe / Danger 双模式和 GNN

普通状态下，系统处于 Safe 模式，使用常规分层重规划。

当传感器读数太小或当前位置 SDF 距离太低时，系统进入 Danger 模式。

Danger 模式下使用：

1. `VisibilityGraph`
   从障碍物表面提取关键顶点，判断顶点之间是否可见。

2. `GNNPlanner`
   在可见图上运行神经动力学活动扩散。

3. `DualModeScheduler`
   用传感器阈值、SDF 阈值和滞回机制决定 Safe/Danger 切换。

这里的 GNN 不是深度学习训练网络。它不是用数据集训练出来的，而是一个基于神经动力学公式的图搜索变体。

答辩时要特别说明：

> 论文 2 原始 GNN 是稠密栅格神经元活动场。本项目采用的是可见图顶点上的工程化变体，称为 vertex-only diffusion，用于危险区域的局部规划增强，而不是替代全局 A*。

## 14. 风险自适应重规划

固定每 0.4 秒重规划有两个问题：

- 环境安全时太浪费计算。
- 环境危险时可能不够快。

项目借鉴论文 3 中“重要性不同就应该分配不同计算资源”的思想，把它工程化为风险自适应间隔。

风险由两部分估计：

- 当前点到障碍物的 SDF 距离。
- 传感器最近读数。

风险高：

- 重规划间隔变短。
- 系统更频繁反应。

风险低：

- 重规划间隔变长。
- 节省计算。

注意：这不是完整 MADDPG，也不是完整 PER，只是使用其思想启发调度策略。

## 15. 故障检测与拓扑重构

论文 4 的思想被拆成三层：

### 15.1 故障注入

模拟某个转子推力下降。例如 severity = 0.3 表示该转子只能输出 30%。

### 15.2 故障检测

检测规则包括：

- 速度突变持续存在。
- 实际位置和期望位置偏差过大。
- 控制量长期接近饱和。

只要任一规则触发，就记录故障。

### 15.3 拓扑重构

检测到故障后，系统在候选队形中选择代数连通度较好的拓扑。

代数连通度通常记作 λ₂。可以通俗理解为：

> λ₂ 越大，编队图越“连得牢”，故障后越不容易散。

项目会自动选择较优队形，并用平滑切换过渡过去。

## 16. Python 与 C++ 两套实现

Python 是主线研究实现，模块完整、测试较多、可视化丰富。

C++ 是性能验证和移植版本，当前已同步：

- 编队动力学。
- 控制器。
- A* / Hybrid A*。
- 在线重规划。
- 风险自适应间隔。
- FIRI 工程化精修。
- GNN Danger 可见图变体。
- 故障注入、检测和拓扑重构。
- SVG/PNG 输出。

仍然需要注意：

- C++ 的事件日志比 Python 简化。
- Python 有 Plotly 交互式 HTML。

## 18. 输出图怎么看

常见输出在 `outputs/`：

- `trajectory_3d_interactive.html`：交互式 3D 轨迹。
- `trajectory_3d.png`：静态 3D 轨迹。
- `trajectory_xy.png`：俯视图。
- `error_realtime_3d.png`：三轴误差。
- `error_statistics.png`：误差统计。

看图时重点关注：

1. leader 是否按任务航点前进。
2. follower 是否保持队形。
3. 轨迹是否穿过障碍物。
4. 在线重规划点是否只在局部出现，而不是替代任务航点。
5. 故障后队形是否发生合理调整。

## 19. 项目创新点总结

可以从五个层次讲：

1. **多层安全链路**
   离线规划、在线重规划、APF 实时避障、FIRI 精修共同工作。

2. **Safe/Danger 双模式**
   安全区使用常规规划，危险区切换到 GNN 可见图增强。

3. **任务航点与局部重规划分层**
   修复在线路径覆盖任务目标的问题，保证执行语义清晰。

4. **风险驱动重规划**
   根据环境危险程度动态调整计算频率。

5. **容错拓扑重构**
   从转子故障注入到检测，再到拓扑重构，形成闭环。

## 20. 一分钟答辩版介绍

本项目面向室内复杂环境下的多无人机编队飞行问题，构建了从四旋翼动力学、编队控制、障碍物建模、路径规划、在线重规划到容错拓扑重构的完整仿真框架。系统采用 leader-follower 编队结构，结合 A*、Hybrid A*、D* Lite、FIRI 凸走廊和改进 APF 实现安全路径生成与实时避障。在危险区域，项目引入 Safe/Danger 双模式调度，并使用 GNN 可见图变体增强局部规划能力；在环境风险变化时，使用风险自适应重规划间隔提高响应效率；在执行器故障情况下，通过状态异常检测和拓扑重构维持编队鲁棒性。项目同时提供 Python 主线实现和 C++ 移植子集，并通过多场景测试和可视化结果验证算法效果。

## 21. 中期新版一分钟介绍

中期版本在原有多无人机编队仿真框架上，补齐了“编队安全裕度如何进入规划决策”的问题。传统 leader-only 路径规划只保证领航机不碰撞，但从机在 diamond、triangle 等队形下会有横向包络，窄通道中仍可能擦碰。因此项目新增 formation-aware clearance 策略模块，把从机包络、队形宽度和局部通道宽度一起纳入判断；同时增加前瞻窗口，在进入狭长通道或大角度转弯前提前触发队形变换。如果局部窗口把前方误判为死路，系统会尝试 RRT escape 候选路径，并把 attempt、accepted、failed 写入事件日志。最后通过消融报告对比 leader-only、普通 formation-aware 和 lookahead adaptive 三种模式，使改进效果不只停留在可视化，而是有 CSV、Markdown 和图片证据支撑。

答辩时可以用一句话概括：本项目的中期创新不是单个规划器，而是把从机安全裕度、提前队形调控和 RRT 前瞻逃逸串成了可运行、可消融、可报告的闭环。
