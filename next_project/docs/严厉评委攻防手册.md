## 1. 总体攻防原则

### 1.1 先承认边界，再给证据

严厉评委最常见的攻击方式是让你过度承诺，然后抓住一个没有完成的细节反杀。防守顺序必须固定：

1. 明确这是仿真系统，不是实机闭环系统。
2. 明确 Python 主线、C++ 可运行子集、Web/dynamic replay 的能力边界不同。
3. 明确哪些是完整实现，哪些是工程化变体，哪些是后续工作。
4. 用测试、报告、消融和代码路径证明“当前中期目标已经达成”。

### 1.2 绝对不要说的话

- “这个算法已经证明安全。”
- “C++ 和 Python 完全一致。”
- “RRT escape 一定能找到路。”
- “snap 指标就等价于完整最优控制。”
- “GNN 是深度学习神经网络。”
- “所有碰撞问题都解决了。”
- “这个结果可以直接迁移到真机。”

### 1.3 推荐说法

- “当前是工程可运行验证，不是形式化安全证明。”
- “Python 主线最完整，C++ 侧已对齐报告、事件和关键预设，是可运行子集。”
- “RRT escape 是局部窗口误判时的候选补充，仍要经过 clearance gate。”
- “`trajectory_snap_squared_integral` 是高阶平滑性代理指标，不等于完整 GCOPTER/MINCO。”
- “GNN 在本项目中指神经动力学启发的可见图活动扩散变体，不是训练式深度网络。”

## 2. 总体定位类刁钻问题

### Q1：你这个项目到底是“算法创新”，还是把一堆现成模块拼在一起？

**攻击点：** 质疑项目只是 A*、APF、RRT、报告工具的拼装。

**危险回答：** “我实现了很多算法，所以有创新。”

**建议回答：**

我的创新点不在于声称每个底层算法都是全新提出，而在于把多无人机编队的安全裕度从“执行后检查”前移到“规划决策”中。具体来说，普通 leader-only planner 只保证领航机轨迹无碰撞，但从机有队形包络，会在窄通道和转弯处发生擦碰。本项目把 `formation-aware clearance`、前瞻窗口队形调整和 RRT escape 候选机制组合成可记录、可消融、可报告的安全调控链路。

这不是简单堆模块，因为每个模块都有明确事件输出：队形调整事件、前瞻阻塞事件、RRT attempt/accepted/failed、碰撞事件和路径指标。最终可以通过 leader-only、formation-aware、lookahead adaptive 三种模式做消融。

**证据路径：**
- `core/formation_adaptation.py`
- `simulations/obstacle_scenario.py`
- `experiments/ablation.py`
- `outputs/ablation_rrt_dual_channel/report.md`
- `outputs/ablation_formation_maze_stress/report.md`
- `docs/中期验收记录.md`

第二，实验证据：`rrt_dual_channel_online` 和 `formation_maze_stress_online` 已生成消融报告，能记录前瞻阻塞、RRT escape 接受和硬碰撞结果。

第三，工程证据：Python 主线和 C++ 可运行子集都能输出报告，C++ 支持关键预设和编队调控元数据。

第四，文档证据：能力矩阵、技术文档、中期验收记录、答辩索引都已统一口径，明确边界。

### Q3：你的系统看起来很大，是否每个模块都真正完成了？

**攻击点：** 用“大而不精”攻击系统可信度。

**建议回答：**

不是每个方向都做到了论文级完整复现，我会分层说明：

- 已作为中期主线完成：编队感知安全裕度、前瞻队形调整、RRT escape、消融报告、中文结果报告。

我的答辩策略不是说“所有模块都完全成熟”，而是说“中期目标主线已经闭环，边界已经文档化”。

## 3. 理论与算法类刁钻问题

### Q4：formation-aware clearance 的数学含义是什么？不是拍脑袋加大障碍半径吗？

**攻击点：** 质疑安全裕度只是调参。

**危险回答：** “我把障碍物膨胀了一点。”

**建议回答：**

不是单纯调大障碍物半径，而是把编队包络映射到路径安全判定中。leader-only 判定只检查：

```text
d(leader_path, obstacle) >= margin
```

formation-aware 判定把从机相对 leader 的最大横向/径向偏移纳入安全裕度：

```text
d(leader_path, obstacle) >= robot_radius + obstacle_margin + formation_envelope_radius
```

其中 `formation_envelope_radius` 与当前队形有关，宽队形在狭窄通道中需要更大 clearance；如果 clearance 不满足，就触发队形调整或拒绝路径。它的价值在于：规划时就考虑从机包络，而不是碰撞后再解释。

**证据路径：**
- `core/formation_adaptation.py`
- `simulations/obstacle_scenario.py`
- `tests/test_formation_adaptation.py`
- `tests/test_obstacle_scenario.py`

### Q5：你如何证明这个 clearance 一定安全？

**攻击点：** 要求形式化证明。

**危险回答：** “一定安全，因为我测试过。”

**建议回答：**

我不能把当前工程原型说成形式化安全证明。当前的 safety 是工程级 gate：对离散路径采样、障碍物距离、队形包络和裕度进行约束，能减少从机包络碰撞风险，但不等于连续时间、连续空间的严格可达性证明。

如果评委要求严格证明，后续需要引入连续时间走廊约束、控制屏障函数或可达集分析。当前中期答辩中，我能证明的是：该模块已经被独立化、可测试、可消融，并且在压力场景中能把 leader-only 漏掉的从机安全问题显式暴露出来。

### Q6：你为什么不用 MPC 或控制屏障函数，非要做这个工程化 clearance？

**攻击点：** 质疑方法不够高级。

**建议回答：**

MPC 和 CBF 是更强的安全控制框架，但当前项目的中期目标是建立可运行、可消融、可报告的多无人机编队仿真闭环。formation-aware clearance 的优势是：

- 计算轻，能嵌入现有 A*/RRT/重规划流程。
- 易解释，能直接输出阻塞原因和队形调整事件。
- 易做消融，能与 leader-only planner 做公平对比。

我没有否认 MPC/CBF 的价值，而是把它们放在后续深化方向。当前阶段先把“编队包络进入规划判定”这件事做成稳定工程闭环。

### Q7：前瞻窗口提前队形变换，会不会导致过度保守？

**攻击点：** 质疑提前收缩损失效率。

**建议回答：**

会有过度保守的可能，所以系统不是无条件收缩，而是由前瞻窗口中的通道宽度、转弯角度、局部阻塞和队形包络共同触发。报告会记录队形调整事件，后续可以统计路径长度、耗时、碰撞和重规划次数之间的权衡。

我的防守点是：窄通道中“晚变换”的代价是碰撞或重规划失败；“提前变换”的代价主要是局部路径长度或队形保持误差。中期目标优先保证安全可解释，再通过消融报告评估效率损失。

### Q8：RRT escape 是不是给算法开了外挂？局部规划失败就随便找条路？

**攻击点：** 质疑 RRT escape 破坏原规划逻辑。

**危险回答：** “RRT 能找到路，所以直接用。”

**建议回答：**

RRT escape 不是无条件替换路径，而是候选生成器。触发条件是局部前瞻窗口出现阻塞或假死路风险；生成候选路径后仍要经过 clearance gate 和路径接受逻辑。报告中会记录 `rrt_escape_attempt`、`rrt_escape_accepted` 和 `rrt_escape_failed`，所以它不是黑箱覆盖。

它解决的是局部窗口在大角度转弯或双通道场景中可能误判“前方死路”的问题。只靠窗口采样看不见绕行通道时，引入全局随机树候选是合理补充。

**证据路径：**
- `maps/rrt_dual_channel_escape.json`
- `maps/formation_maze_stress.json`
- `experiments/scenario_registry.py`
- `outputs/ablation_rrt_dual_channel/summary.csv`

### Q9：RRT 本身不最优，你凭什么说路径最优？

**攻击点：** 抓住“最优”措辞。

**危险回答：** “因为 RRT 找到的路径最短。”

**建议回答：**

我不会把当前结果表述为数学全局最优。报告里的“最优”是当前预设、当前候选规划器、当前指标权重下的相对最优。指标包括路径长度、最大速度、最大加速度、碰撞、航点到达、重规划事件、jerk/snap 平方积分等。

更准确的说法是：在给定候选集合和实验配置下，系统选择综合评分最好的路径；不是证明全局最优控制解。

### Q10：为什么从 jerk 平方积分换成更高阶 snap 平方积分？你真的懂四旋翼动力学吗？

**攻击点：** 用动力学阶次攻击指标选择。

**危险回答：** “snap 越高级越好。”

**建议回答：**

四旋翼不是简单二阶质点系统。位置轨迹的 jerk 能反映一部分平滑性，但四旋翼姿态和控制输入变化通常与更高阶导数相关，minimum snap 在四旋翼轨迹生成中更常见。因此我保留 `trajectory_jerk_squared_integral` 作为辅助指标，把 `trajectory_snap_squared_integral` 作为更贴近四旋翼轨迹平滑性的代理。

但我也明确边界：当前是工程化指标提取和轻量轨迹后处理，不是完整 GCOPTER/MINCO 求解器，也不是严格最优控制证明。

**证据路径：**
- `experiments/metrics_extractor.py`
- `docs/技术文档.md`
- `docs/中期规划书.md`

### Q11：你用 snap 指标，但没有真正做 minimum snap 优化，这不是概念偷换吗？

**建议回答：**

不是偷换，因为文档中已经明确区分“指标”和“优化器”。`trajectory_snap_squared_integral` 当前用于评价轨迹高阶平滑性；`minimum_jerk` 或 `min_snap_proxy` 是轻量工程原型，不宣称等价于完整 minimum snap trajectory optimization。

如果要继续深化，我会把下一阶段目标设为：引入带时间分配和动力学约束的多项式轨迹优化，再把 snap 从评价指标推进到优化目标。

## 4. 动力学与控制类刁钻问题

### Q12：你的四旋翼模型是否考虑真实电机延迟、空气动力、饱和和传感器噪声？

**攻击点：** 用真实系统复杂性压仿真模型。

**建议回答：**

当前模型是仿真级四旋翼动力学模型，重点支持轨迹跟踪、编队控制、规划闭环和故障注入，不宣称覆盖全部真实空气动力和硬件延迟。项目中有转子故障、风场扰动、控制误差和部分传感器抽象，但还不是实机级高保真模型。

答辩中我会把它定位为“算法与系统验证平台”，而不是“可直接部署的飞控系统”。

### Q13：PID、SMC、反步控制都放进去，是不是没有主线？

**攻击点：** 质疑控制器堆叠。

**建议回答：**

控制器不是主创新点，而是仿真平台的可切换执行层。PID 提供稳定基线，SMC 和反步控制用于鲁棒性和非线性控制对比。项目主线在编队规划和安全调控，不依赖某一个控制器才能成立。

答辩时我会避免把控制器数量当作创新点，而是说明它们用于支撑不同场景下的执行验证。

### Q14：如果控制器跟踪误差很大，你的路径安全还有意义吗？

**攻击点：** 连接规划与控制误差。

**建议回答：**

这是合理质疑。路径安全判定本身假设执行误差在一定范围内，所以报告中需要同时输出路径参数和飞行误差。如果跟踪误差超过安全裕度，即使规划路径无碰撞也可能发生碰撞。

当前系统通过误差统计、碰撞日志、航点到达事件和安全裕度记录来暴露这个问题。后续可以把跟踪误差上界并入 clearance margin，让规划裕度随控制误差自适应放大。

### Q15：你说容错拓扑重构，λ2 就能说明编队可靠性吗？

**攻击点：** 质疑代数连通度指标单一。

**建议回答：**

λ2 只能说明拓扑连通性和信息传播能力的一部分，不等于完整任务可靠性。它适合用于故障后候选拓扑的快速筛选，但还需要结合队形误差、碰撞风险、执行能力和任务完成度。

所以我不会说“λ2 高就一定最好”，而会说“λ2 是拓扑重构中的一个可解释指标，用来避免故障后通信或协同结构断裂”。

## 5. 规划与避障类刁钻问题

### Q16：A*、Hybrid A*、D* Lite、RRT、APF、FIRI、GNN 都出现了，你的主规划器到底是什么？

**攻击点：** 质疑系统杂乱。

**建议回答：**

主线不是“单一规划器崇拜”，而是分层规划结构：

- 全局或栅格路径：A*/Dijkstra/Hybrid A* 等提供初始路径。
- 动态更新：D* Lite 或 WindowReplanner 处理局部变化。
- 局部避障：APF 和 Safe/Danger 模式处理短时风险。
- 安全过滤：formation-aware clearance 决定路径是否可被编队接受。
- 逃逸候选：RRT escape 在局部窗口误判死路时提供候选路径。
- 报告比较：planner comparison 只在同一预设和指标下给相对最优解释。

### Q17：APF 有局部极小值问题，你怎么解决？

**建议回答：**

APF 的局部极小值问题确实存在，所以项目没有把 APF 当成唯一规划器。APF 只作为局部避障和前馈修正的一部分；当局部路径不可接受时，系统会触发重规划、GNN/Danger 模式或 RRT escape 候选。报告也会记录阻塞和回退原因。

换句话说，APF 在这里是局部反应层，不是完整全局可达性保证。

### Q18：FIRI 在你项目里是严格论文复现吗？

**建议回答：**

不是严格论文级完整复现，而是工程化走廊精修模块。它的定位是给路径提供更安全的凸走廊或近似安全区域，不是中期创新主线。答辩时我会主动说明这个边界，避免把工程化变体说成完整理论复现。

### Q19：GNN 是图神经网络吗？你有训练数据吗？

**危险回答：** “是图神经网络，但是没有训练。”

**建议回答：**

这里的 GNN 不是深度学习里的 Graph Neural Network，而是神经动力学启发的规划模型名称，项目中工程化为可见图活动扩散和 Safe/Danger 双模式调度。因此不需要训练数据集，也不应该把它说成训练式图神经网络。

如果老师追问，我会强调命名容易混淆，所以答辩中会主动解释 GNN 的含义。

## 6. 实验与数据可信度刁钻问题

### Q20：你的地图是不是为了你的算法特意设计的？

**攻击点：** 质疑场景过拟合。

**建议回答：**

`rrt_dual_channel_online` 和 `formation_maze_stress_online` 的确是压力测试地图，不是随机自然分布地图。它们的作用是暴露 leader-only 在狭窄通道、大角度转弯和局部假死路中的缺陷。压力场景用于机制验证是合理的，但我不会把它们说成充分统计泛化。

后续要增强泛化，需要增加随机地图、多 seed、多障碍密度和更长仿真时间的 full 实验。

### Q21：为什么只看两个压力场景？样本太少怎么证明有效？

**建议回答：**

中期阶段的重点是机制闭环和可复现消融，不是大样本统计论文。两个压力场景分别覆盖双通道假死路和迷宫式编队压力，已经能验证前瞻队形调整和 RRT escape 是否被触发、是否被记录、是否能减少硬碰撞。

我承认样本量不足以证明泛化性能，后续会增加随机地图和多 seed 统计。

### Q22：硬碰撞为 0 是不是因为你把碰撞判定放宽了？

**攻击点：** 质疑指标作弊。

**建议回答：**

需要区分硬碰撞、clearance 阻塞和软风险。硬碰撞为 0 不代表没有风险，而是代表在当前碰撞判定下没有发生实体碰撞。为了避免掩盖风险，报告同时输出 clearance blocked、formation adaptation events、RRT attempt/accepted、路径长度、速度、加速度和误差。

如果评委担心阈值放宽，可以现场说明碰撞半径、障碍膨胀和 formation envelope 的配置来源，并承认后续需要做阈值敏感性分析。

### Q23：你说报告自动生成，报告内容会不会只是美化，不具备科学价值？

**建议回答：**

报告不是单纯美化，它把仿真结果结构化为可复核证据：路径图、飞行参数、误差统计、碰撞事件、航点到达、规划器耗时、重规划事件和最优性解释。报告中的数值来自 `sim_result.json`、`metrics.json` 和消融 `summary.csv`。

它的价值是减少人工挑图和选择性展示，让每次仿真都有固定输出格式。

**证据路径：**
- `experiments/result_reporter.py`
- `experiments/report_writer.py`
- `experiments/report_cpp_results.py`
- `experiments/run_cpp_report.py`

### Q24：你做 planner 对比时，是否保证每个 planner 独立执行完整轨迹？

**攻击点：** 抓 benchmark 语义。

**建议回答：**

要分清两个层面。Python 的 planner comparison 和消融实验用于给定预设下的相对对比；C++ dynamic replay 的某些 summary 更偏逐帧规划开销观察，不应被解释成严格独立 benchmark。这个边界已经在后续实施映射和能力矩阵中说明。

答辩中如果展示 planner 对比，我会优先使用 Python 主线消融报告，不把 dynamic replay 的 summary 过度解释为完整横向 benchmark。

### Q25：Python 和 C++ 到底对齐到什么程度？你是不是只改了字段名？

## 8. 代码质量与测试类刁钻问题

### Q30：你的测试很多，但是否真的测到了关键行为？

**攻击点：** 质疑测试只是覆盖表面。

**建议回答：**

测试分三类：

- 单元行为：例如 formation adaptation、metrics extractor、报告字段。
- 工作流行为：例如 experiment workflow、C++ report workflow。
- 回归行为：例如 cross-line regression 和 result schema。

我承认测试仍有边界：随机地图泛化、完整 C++ runtime parity、实机误差和连续时间安全证明还没有完全覆盖。当前测试能证明中期闭环可运行和关键事件可追踪，但不是最终产品级验证。

### Q31：你如何证明报告里没有空字段，比如路径长度、速度、加速度、jerk、snap？

**建议回答：**

针对这个问题已经补了 `metrics_extractor` 和 C++ 报告测试。报告侧会从轨迹采样或路径数据计算 `trajectory_path_length`、`trajectory_max_speed`、`trajectory_max_acceleration`、`trajectory_jerk_squared_integral` 和 `trajectory_snap_squared_integral`。如果 C++ 缺少 Python 的 `planned_trajectory`，报告会走兼容路径并保留缺失说明。

**证据路径：**
- `experiments/metrics_extractor.py`
- `tests/test_metrics_extractor.py`
- `tests/test_cpp_result_reporting.py`

### Q32：你是不是为了让测试通过，把仿真时间缩短到失真？

**建议回答：**

quick 模式用于快速回归，目的不是替代完整实验。它验证字段、事件、流程和基本行为是否可运行。中期收口报告中记录了 quick 与 full 的边界，后续论文型实验需要使用更长仿真时间、多 seed 和更完整场景。

答辩中如果展示 quick 结果，要说清楚它是回归证据，不是最终统计性能。

### Q33：如果老师看代码，最容易被质疑的点是什么？

**建议回答：**

我会主动说明四个风险点：

1. Python 主线和 C++ 子集不是完全等价。
2. dynamic replay 不等于完整多机动力学 benchmark。
3. 轨迹优化仍是轻量原型。
4. 部分规划器是工程化变体，不是完整论文复现。

主动说明这些边界，反而能提高可信度。

### Q34：`formation_adaptation.py` 如果只是策略函数，为什么说它是算法模块？

**攻击点：** 质疑“模块化”只是把代码换了文件。

**建议回答：**

它不是单纯换文件，而是把编队安全判定从场景主循环里抽出来，形成明确输入和输出。输入包括当前队形、从机包络、通道或障碍物距离、规划路径和触发上下文；输出包括是否阻塞、推荐队形、原因标签和事件记录字段。这样主循环只负责调用和执行，不再把“如何判断编队安全”写死在仿真过程里。

模块化的价值是三点：可单测、可替换、可做消融。后续如果换成 CBF 或 MPC 约束，也可以沿用同一层接口。

### Q35：如果我把 `formation_aware_lookahead_adaptive` 关掉，系统是否还能跑？

**建议回答：**

应该能跑。消融设计的目的就是让 leader-only、普通 formation-aware 和 lookahead adaptive 在同一场景下可切换。关闭 lookahead adaptive 后，系统会退化到不使用前瞻队形调整和 RRT escape 的基线或普通编队感知模式。这样才能比较新增机制带来的差异。

**证据路径：**
- `experiments/ablation.py`
- `experiments/run_ablation.py`
- `tests/test_experiment_workflow.py`

### Q36：事件日志是否会影响仿真行为？会不会为了报告而改变算法？

**攻击点：** 质疑报告逻辑污染仿真。

**建议回答：**

事件日志原则上是旁路记录，不应反向改变仿真决策。真正影响行为的是规划器、clearance gate、队形调整策略和路径接受逻辑；报告生成器只读取 `sim_result.json`、`summary.csv` 和事件列表。答辩时可以按数据流说明：仿真生成结果，metrics extractor 提取指标，report writer 渲染报告，方向是单向的。

如果后续发现报告字段反向进入控制逻辑，那会是代码设计问题，需要拆开。

### Q37：C++ 输出字段增加了，怎么证明不是“假数据”？

**攻击点：** 质疑 C++ 字段只是占位。

**建议回答：**

要分字段回答。路径、飞行参数、碰撞、航点和规划耗时来自 C++ 仿真输出或报告计算；编队调控元数据和 `formation_adaptation_events` 用于对齐 Python 报告语义。对于仍属于展示或子集的字段，文档已经标注边界，不把它们说成完整 Python 等价。

防守重点不是说“所有字段都完全等价”，而是说“哪些字段可作为仿真证据、哪些字段用于展示、哪些字段仍是边界”，并且这些边界写进了能力矩阵和后续实施映射。

### Q38：如果某个测试只验证字段存在，不验证行为，你怎么解释？

**建议回答：**

字段存在测试只能证明 schema 和报告兼容，不能证明行为正确。这类测试的价值是防止输出结构断裂；行为正确还需要场景运行测试、消融结果和指标变化共同证明。

当前项目已经有字段测试、工作流测试和场景消融，但我承认还需要更强的运行时对齐测试，尤其是 C++ 与 Python 的行为一致性、随机地图和多 seed 统计。

### Q39：你如何快速定位一次碰撞的原因？

**建议回答：**

按四层排查：

1. 看 `collision_log`，确认碰撞时间、对象和位置。
2. 看同一时间附近的 `formation_adaptation_events` 和 `planning_events`，判断是否发生前瞻阻塞、RRT escape 或路径拒绝。
3. 看轨迹误差和速度/加速度，判断是规划包络不足还是控制跟踪误差导致。
4. 回到地图和队形配置，检查通道宽度是否小于当前队形 envelope。

这样能把碰撞归因到 leader-only 包络不足、队形变换过晚、局部窗口误判、控制误差或参数阈值过紧。

### Q40：如果老师现场问“这个字段在哪一行生成”，你如何回答？

**建议回答：**

不要硬背行号。应该回答生成链路：Python 侧先看 `simulations/obstacle_scenario.py` 的仿真主循环和事件记录，再看 `experiments/metrics_extractor.py` 的指标提取，最后看 `experiments/result_reporter.py` 或 `report_writer.py` 的报告渲染。C++ 侧先看 `cpp/src/warehouse_main.cpp` 或 `cpp/src/dynamic_main.cpp` 输出 JSON，再看 `experiments/report_cpp_results.py` 生成报告。

如果老师要求精确行号，可以现场搜索字段名，而不是凭记忆编。

## 9. 追问链训练

### 追问链 A：创新性

**评委：** 你这个 formation-aware clearance 不就是把半径调大吗？

**答：** 如果只是统一调大半径，那确实只是保守调参。本项目的区别是安全裕度随队形包络变化，并且会触发队形调整和事件记录。宽队形、窄队形、转弯窗口下的判定不同，因此它是编队状态相关的安全 gate。

**评委追问：** 那为什么不直接一直用最窄 line 队形？

**答：** 一直用 line 会牺牲覆盖效率和编队任务表达。项目的目标不是固定最保守队形，而是在通道风险出现前自适应调整，风险解除后仍可恢复更合适队形。

**评委再追问：** 你证明这样更优了吗？

**答：** 我不声称全局最优。当前证明的是在压力场景中，它比 leader-only 更能暴露并缓解从机安全裕度问题；最优性仅在当前指标权重和候选模式中相对成立。

### 追问链 B：RRT escape

**评委：** 局部窗口走不通，你就用 RRT 绕，这是不是补丁？

**答：** 它是显式设计的候选路径机制，不是隐藏补丁。局部窗口在大角度转弯中容易误判死路，RRT escape 负责补充全局候选路径，并且所有 attempt、accepted、failed 都写入事件日志。

**评委追问：** RRT 找到的路可能很差。

**答：** 是的，所以候选路径仍要经过 clearance gate 和路径接受逻辑，不是找到就用。报告也会输出路径长度、平滑性和规划耗时，让差路径暴露出来。

### 追问链 C：轨迹优化

**评委：** 你先说 jerk，后来说 snap，是不是理论不稳定？

**答：** 这是中期过程中的修正。最初用 jerk 作为平滑性指标，后来意识到四旋翼不是二阶质点系统，因此把 snap 平方积分作为更合适的高阶平滑性代理，同时保留 jerk 作为辅助指标。这是对模型认识的修正，不是概念摇摆。

**评委追问：** 那你完成 minimum snap 了吗？

**答：** 没有完成严格 minimum snap 优化器。当前完成的是 snap 指标输出和轻量优化原型，后续才会推进到更严格的轨迹优化。

### 追问链 D：C++ 对齐

**评委：** 你说 C++ 对齐，但又说不是完全等价，这不是自相矛盾吗？

**答：** 不矛盾。对齐分层：报告字段、预设入口、事件结构和部分安全语义已经对齐；完整算法能力没有完全等价。我的表述是“C++ 可运行子集已对齐中期报告需求”，不是“C++ 完整复制 Python 主线”。

### 追问链 E：实验可信度

**评委：** 两个地图能说明什么？

**答：** 它们能说明机制是否成立，不能说明统计泛化。双通道图验证局部假死路和 RRT escape，迷宫压力图验证狭窄通道、转弯和队形调整。泛化结论需要后续多 seed 和随机地图。

## 11. 高频狠问题短答库

### Q：你这个项目最大短板是什么？

最大短板是真机验证和严格理论证明还没有完成。当前是仿真验证平台，优势是链路完整、可复现、可报告；不足是实机噪声、硬件延迟、连续时间安全证明和大样本统计还需要后续补强。

### Q：如果评委只允许你保留一个创新点，你保留哪个？

保留 formation-aware clearance 与前瞻队形调整。因为它直接针对多无人机编队区别于单机规划的核心问题：leader 安全不代表从机安全。

### Q：你的工程工作量是否掩盖了理论薄弱？

工程工作量不是用来掩盖理论，而是把理论假设变成可运行、可测量、可审计的流程。理论边界我会主动说明；当前贡献是把编队安全裕度、在线重规划和报告验证做成闭环。

### Q：为什么不直接用成熟仿真器，比如 AirSim 或 Gazebo？

成熟仿真器适合高保真环境和传感器仿真，但本项目重点是算法链路可控、指标可提取、消融可复现。自建轻量框架便于快速修改规划、队形和报告字段。后续实机或高保真验证可以接入 Gazebo/AirSim。

### Q：你怎么保证不是“调参调出来”的？

看消融和事件链。如果只是调参，不会产生清晰的 `lookahead_reference_blocked`、`rrt_escape_attempt`、`rrt_escape_accepted`、`formation_adaptation_events` 等机制性证据。项目不是只比较最后碰撞数，而是记录触发原因和决策过程。

### Q：如果现场复现实验失败怎么办？

先区分失败类型。如果是环境或 C++ 构建问题，使用已记录的测试结果和输出目录作为证据；如果是 Python quick 实验失败，查看 traceback、配置 preset 和输出目录。答辩时不能硬说通过，要说清楚最近一次验证命令、结果和当前失败原因。
